Scope: 82 files, +9,858 / −4,131 Base: d79aa633 (develop) Head: 68793991 Reviewed: 2026-06-11

TL;DR

The feature's central promise — that what you configure on the track page is what runs — is broken in four independent ways. The UI writes to the track's forked pipeline and to the resolver's override layers, but the runtime still reads pipeline 'default' and the raw stage_state/stage_mode columns. Prompt edits, agent assignments, template-inherited mode/include settings, and harness/model overrides all display as "saved / effective" while having zero effect on actual runs.

Findings 1, 2, and 5 are facets of the same architectural gap — the UI surface was cut over in Phase 6, but the runtime was never wired to the resolver — and likely want one coordinated fix rather than three patches.

Critical 1. Stage prompt/agent/gate edits are written to the track's fork, but the runtime always reads pipeline 'default' — edits silently never reach execution

Location

apps/api/src/services/stage-prompt-assembler.service.ts:132

Detail

The track page's tabs persist stage prompt, agent-binding, and quality-gate edits via PUT /api/pipeline-stages/:id?pipelineId=<track-fork> (prompts tab :416, agents tab :315). Runtime prompt assembly never receives a pipeline id: agentic-step-handler.ts:1255 calls assemble() without one, so the assembler falls back to 'default' (line 132) for the stage prompt, agent binding, skills, and pipeline prompt. Nothing syncs the fork back to 'default'. The PR also deletes the old /pipeline/config page — the only UI that edited the 'default' pipeline — so post-merge there is no surface whose stage-content edits reach execution.

Failure scenario

User edits the implement stage's action prompt on the track page and gets a success toast. They run a story. The agent executes with the old prompt, because assembly read the untouched 'default' pipeline. There is no remaining UI to edit the pipeline that actually runs.

Verifier verdict: CONFIRMED — fork write path, default-only read path, absence of any sync, and deletion of the old surface all verified in code.

Detail

Cleanup 10. createTrack discards its complete POST response and awaits a full re-fetch of the N+1 tracks endpoint — doubling create-dialog latency

Location

apps/ui/src/app/features/pipeline/track-config.service.ts:128

Detail

The POST response already contains the created track with stageConfig and server-assigned displayOrder, yet createTrack awaits loadForWorkspace(workspaceId, true) before returning. That endpoint (GET /:id/tracks, routes/tracks.ts:169) is itself an N+1: one pipeline_stages query per track via withStageConfig, trivially batchable with WHERE pipeline_id IN (...). Verifier verdict: CONFIRMED on both halves.

Recommendation

Do not merge as-is. Findings 1, 2, and 5 amount to "the runtime was never wired to the resolver": the new configuration surface looks authoritative but is partially or wholly inert, and the old surface that was authoritative has been deleted. The per-phase test coverage pins the resolver's behavior, not the executor's consumption of it — which is exactly the seam where these bugs live. Suggested order: wire runtime gating and prompt assembly to the track/resolver (1, 2, 3), close the legacy-backfill gap (4), then either wire or visibly de-scope the override layers (5).
