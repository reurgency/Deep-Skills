How /deep-code-review --multi-agent Works Now

The 8-dial efficiency redesign (2026-06-12), validated blind on PR #65 ("eco v4"): a five-stage pipeline — scripted setup, eight model-tiered single-lens finders, orchestrator dedup, severity-gated tiered verification, script-assembled synthesis. Measured: 6.51M weighted tokens · 23% of a 5-hour limit (~5% weekly) · 46 minutes — both catchable Criticals found, 29% cheaper than Eco V2, a third of Combo V1.

The pipeline in detail

Five stages. Read-only agents throughout — the skill never edits source, never reads .env/secrets, never passes the implementation transcript to any agent.

Stage 0 · Scripted setup scripts

deterministic — costs ~zero LLM tokens

Diff pre-chunking: the resolved diff is split per-file into chunk files plus an index (path, +/− counts) — finders read chunks per-file as their lens requires, never the monolithic diff. Pre-pass in parallel: lint · typecheck · the test suite as one combined invocation launch alongside the fleet (finders never consume their output; it joins at synthesis). Base-vs-head test attribution is scripted: two worktrees, both suites concurrently, normalized outputs diffed. Verifier fact pre-harvest: before Stage C, a script collects each finding's cited evidence lines, symbol definitions, and caller lists so verifiers don't burn their first calls re-finding what grep finds.

chunk index + chunks + scope line + plan (never the implementation transcript)
▼

Stage A · Finder fleet — 8 independent single-lens passes from fab solo

one parallel batch · cap 8 · read-only · model set explicitly per launch

Each finder is a fresh agent with exactly one lens — never two. The four Blocker/Major-producing lenses run on the main tier; the three structural lenses on the mid tier; the pattern-matching quality sweep on the cheap tier.

Stage B · Dedup + decomposition

orchestrator (main session) — no new agents
Merge — when one fix resolves both reports
One root cause seen by several passes becomes ONE finding carrying multiple dimension tags, keeping the strongest evidence from each.
Split — when one candidate bundles independent fixes
N independent fixes → N findings. A defect cited as evidence inside another candidate gets filed separately too. Test: how many independent fixes does this describe?
deduped findings → severity gate
▼

Stage C · Severity-gated, tiered adversarial verification

one fresh verifier per finding · single streamed wave · read-only
main tier sev ≥6 — briefed to refute
  • Burden of proof on the finding — the verifier attacks the evidence against the actual code.
  • Sub-claims verified independently; severity recalibrated by counting user-reachable trigger paths.
  • Fixed machine-readable verdict format (VERDICT / SUB-CLAIMS / SEVERITY / RATIONALE).
confirmed → report plausible → kept, marked refuted → appendix
cheap tier sev 5 — demotion-checking
  • Calibration: every sev-5 verification ever run ended in demotion or refutation — a cheap model handles that honestly.
  • If a cheap verifier raises severity to ≥6, a main-tier verifier re-checks before synthesis accepts it.
sev ≤4 (Nit) → ships unverified
Verifying the nit tail once consumed ~a quarter of a run for zero changed verdicts.
  • Report states the floor, so "unverified" is interpretable.
  • Synthesis may promote a load-bearing sev-4 into verification (stated why).
  • Never lower the floor mid-run to "be thorough".
unverified → nit table
structured verdicts · unverified nit tail · pre-pass results join here
▼

Stage D · Script-assembled synthesis → ONE report

script builds the artifacts · the orchestrator writes only judgment

findings.json, severity rollup, verification funnel line, nit table, and certificate scaffold are assembled by script from the structured verdicts. The model writes what only a model can: evidence-rule enforcement (no chain, no last-mile finding), severity reconciliation with stated rationale, merge/split calls, and per-finding prose for sev ≥5 — in-session, never delegated to a smaller model.

findings.json

certificate.md
pass/fail — pass only when no unresolved Blockers

Stage E · Triage + fix-phase routing from fab deep

human in the loop

Blockers auto-route to the fix-phase (user informed, not asked). Everything else is triaged finding-by-finding: fix → fix-phase appended to the plan for /deep-implement · defer → the plan's Deferreds ledger · reject → recorded as "rejected by user", never silently dropped.

--mega — the thorough tier. The same five-stage pipeline with the cost/accuracy dials at maximum: all eight finders and all verifiers on the main tier, quality's sev cap lifted, at roughly double the default's wall clock and rate-limit footprint. The scripted stages stay — determinism trades no accuracy. The Haiku ban holds here too. For release gates and large risky merges; the cost is confirmed with the user before launch.
