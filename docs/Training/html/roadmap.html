The shape of the whole series

Eight cross-cutting decisions

Some decisions aren't owned by any single skill — they recur across all of them. Learn these and the per-skill choices read as instances of a few ideas.

01

One skill, one job

The pipeline is split so no skill does two things — and code-review even splits reviewing from triaging. The clearest evidence is the commit that pulled all routing out of the review run into a separate --triage step.

02

Fresh-agent resumability

Every artifact lets a fresh agent continue from only the artifact + the repo. The plan is a durable spine; implement carries per-phase hand-offs; deep-docs tiers load independently.

03

Independence through fresh eyes

Both reviewers never see the upstream transcript. deep-bug-fix tunes this — continuity for diagnose→fix, a fresh adversarial agent for proof so the fixer can't grade itself.

04

The last mile + evidence

Each skill has its own dialect of the last-mile failure — implement's “green ≠ verified,” review's last-mile lens, bug-fix's “claimed fix,” deep-docs's “stale map.” Every claim cites evidence.

05

Improvements are data, not prose

Recurring bug classes become toggleable directive cards. A SKILL.md is never edited to encode a lesson — that causes bloat, drift, merge conflicts, and no clean on/off.

06

Standalone, two byte-identical files

Each skill self-contains; only the directives/ registry is shared. The two host-agnostic files are copied byte-identical, with a consistency sweep. “Reuse” here means copy, not link.

07

Shift-left

When code-review proves it catches a class only late, prevention is pushed upstream — interaction-edge + re-entry became a plan REQUIRES / plan-review CONFIRMS pair; external-contract became implement construction rules.

08

Natural-language invocation is primary

The series must be drivable on Copilot or Codex, which have no slash-commands. Natural language is primary; --flags are a convenience on top. deep-docs is the reference implementation.

The self-improving loop

Deep-Learn, in more depth

The loop is hub-and-spoke. deep-code-review is the producer — its findings.json is the loop's input. A distiller turns recurring classes into cards, gated by a human. The upstream skills — plan, plan-review, implement — are the consumers that load active cards and prevent the bug earlier. code-review itself carries no card loader: it measures, it doesn't apply.

DLC-001 is a many-to-many class

The seed card touches three skills at once: deep-plan requires the State/Data-Flow Contract, deep-plan-review checks it, deep-implement validates it with a contract probe. One lesson, distributed across the pipeline as roles — exactly what the data-not-prose design makes cheap.

Why never edit a SKILL.md

Encoding a lesson by editing prose causes skill bloat, drift across the spokes, merge conflicts, and — fatally — no clean on/off. A card has a status flip, provenance, and effectiveness telemetry. The skills stop changing; the registry changes.

5th skill · design outline

deep-bug-fix

One job: diagnosis-first remediation of existing behavior under regression risk — root-cause clustering, proof-of-fix, and blast-radius containment that /deep-implement structurally lacks.

Signature failure it hunts: the claimed fix — a change that makes the reproduction stop without addressing the root cause, or that patches one symptom of a cluster and silently leaves its siblings.

Why a dedicated skill (vs re-running implement on findings)?

/deep-implement is forward construction from a trusted spec: known steps, adds code, gates on new failures in changed files. Bug-fixing inverts all four assumptions — the spec is unknown (diagnose first), you modify working code (the highest-risk edit class), findings often share one root cause, and “done” means provably gone, not “the step ran.” Pointing a fresh agent at findings.json and saying “fix these” yields N symptom patches with no proof and no regression guard.

DECISION

Diagnosis before remediation

The finding's recommendation is a hypothesis, not a spec — bug-fix re-diagnoses independently and may reject the suggested fix. (A real finding even hedged: “Register the route or correct the service URL.”)

DECISION

Root-cause clustering

Group distinct findings by shared underlying cause, one fix per cause — a coarser altitude than review-time dedup. One benchmark PR's six findings shared a single cause.

DECISION

Proof-of-fix, two tiers

Default: a static chain-trace proof (reusing the last-mile method). --reproduce: a dynamic red→green test. Both gated by an adversarial proof agent — “the fixer does not mark its own homework.”

DECISION

Regression containment by default

Identify changed symbols, find their callers/dependents, re-run their tests. The danger lives in the dependents — containment looks outward, not just inward (implement validates imports, not dependents).

DECISION

Tuned independence

Diagnose + fix is one agent per cluster (continuity is the point); proof is a fresh adversarial agent returning fixed | unproven | regressed. Keeps the series' DNA exactly where it pays.

DECISION

Triage still decides

Bug-fix is the specialized executor for the accepted set only — it makes no triage decisions. It replaces implement as the fix-phase executor and also runs standalone on a raw bug report / trace / failing test.

Open by design

The design note records its own unresolved questions (round model for re-review, normalized defect shape, whether to keep implement's fix-phase path, regression-scope discovery by grep vs LSP). These are deferred to the eventual /deep-plan pass — not missing rationale, but decisions intentionally left open at this stage.

6th skill · planned

deep-docs

One job: the series' missing orientation layer — context-window-aware documentation of what's been built, structured for Progressive Disclosure so an agent under a token budget loads only the slice its task needs.

Signature failure: the dual failure of AI-facing docs — the stale map (describes intent, not the code as it is) and the bloated dump (must be loaded whole).

DECISION

Three loadable tiers

Tier-0 MAP.md + index (always-loaded, tiny), tier-1 subsystem cards (on touch), tier-2 deep refs (on demand). The structural defense against the bloated dump.

DECISION

Anchored or it's fiction

No prose enters a trusted tier without a resolvable path:line (symbol) anchor, and every anchor is adversarially verified. The defense against the stale map.

DECISION

Symbol-primary anchoring

Verify re-resolves by symbol first, re-snaps the line if it drifted (benign), flags drifted only if the symbol is gone. Line numbers are fragile; content hashes are too sensitive.

DECISION

Exhaustive verify, no sampling

One fresh verifier per card re-resolves every anchor; any drift blocks publish. Symbol-grep is cheap, and the guarantee is binary — one unverified anchor and the tier can't be trusted.

DECISION

Machine-owned artifacts

docs/ai-map/ is 100% machine-owned with a “regenerate overwrites” header. Docs here are an index, not a narrative — humans annotate elsewhere, so a single rendering can't fall out of sync with itself.

DECISION

Quarantine existing docs

Useful READMEs are indexed as external pointers marked “not anchor-verified,” never absorbed into a trusted tier — laundering unverified prose would silently break the anchoring guarantee.

A reflexive observation

deep-docs “recreates Claude Code's own Skills architecture” — tier-0 is an always-loaded index, tier-1 cards load on touch, tier-2 refs on demand: exactly the progressive-disclosure shape a SKILL.md + references/ uses. The orientation layer for agents is, structurally, the same idea as the skill format that produced it.

A note on stage — read rationale at the right altitude

The four built skills' rationale is anchored to live skill files + git history. deep-bug-fix is anchored to its design note and deep-docs to its resolved plan — both have open questions still unresolved by design. When these ship, expect details to firm up; the shape of the decisions above is settled, the specifics may move.
