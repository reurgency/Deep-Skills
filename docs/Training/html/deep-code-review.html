deep-code-review

/deep-code-review independently reviews implemented code the way plan-review reviews a plan: fresh agents, evidence-required findings, a clean report. Its signature concern is the last-mile problem. Review only — never edits source. Triage is a separate, opt-in step.

The frame

Mental model

The implementer is too close to the code to see what it left out. The review runs in fresh agent(s) that receive the diff + the plan + codebase access — but never the implementation transcript.

Its signature concern is the last mile: code where the interface looks correct and the happy path appears connected, but the behavior doesn't make it the last mile — optimistic UI with no real call, swallowed errors, half-wired chains, silently dropped plan scope.

Review ≠ triage

The review autonomously produces findings (all open) and stops. A human decides fix/defer/reject later, under --triage.

Determinism ≠ judgment

Lint/typecheck/tests run as scripts; agents do the reasoning. Compilers are exact at what they do — don't make an LLM do it.

How to use it

The review, module by module

  1. Module 1

    Resolve scope, and state it

    No args: current branch vs the base it was cut from (PR-declared base → merge-base candidates → ask on any ambiguity). PR argument: resolve via gh. Explicit paths: review just those. Before launching anything, print the resolved scope in one line — e.g. Diffing feat/x against develop (working tree included) — 14 files, +812/−214. Never guess a base branch.

  2. Module 2

    Resolve the effort & artifact home

    A deep-* plan exists → artifacts go to that effort's 04-Code-Review/; read the plan in full. No plan → ask for an effort name (default: slugified branch); create the dir + manifest.

  3. Module 3

    Deterministic pre-pass

    Run lint, typecheck, and tests as deterministic tools, never agents. Discover commands from the host project — never from .env/secrets. Results feed the report as findings.

  4. Module 4

    Match situational checks

    Match the diff against the situational-check catalog — surface-specific bug hunts whose bodies load only when the diff matches their trigger. Append each matched body to the brief of the lens it attaches to. Disclose which checks fired in the report header.

  5. Module 5

    The four lenses

    Default: one fresh agent, four lenses — the “would a senior engineer approve this PR” pass. Details below.

  6. Module 6

    The last-mile methodology

    Enumerate promised behaviors, walk each chain hop-by-hop with evidence per hop, hunt the named AI failure modes. Findings without a cited chain are rejected at synthesis.

  7. Module 7

    Findings & report — the review stops here

    Write report.md, findings.json, certificate.md into 04-Code-Review/ — every finding open. No triage decisions, nothing written outside that folder (it never edits the plan). Close with: Review complete — N findings (status: open). Run /deep-code-review --triage…

  8. Module 8

    Triage — the only mode that writes to the plan

    On the latest open findings: Blockers auto-accept (user informed); Major/Minor/Nit → HITL fix/defer/reject. Accepted findings become a fix-phase appended to the plan so /deep-implement can execute it cold.

Module 5, expanded

The four lenses

LENS 1

Correctness

Logic errors, null handling, race conditions, broken error paths.

LENS 2 · the signature

Functional Completeness (Last-Mile)

Every behavior the plan promises, traced hop-by-hop through the full chain — not merely a connected happy path.

LENS 3

Plan Conformance

Everything the plan said is present; no silently dropped scope or undeclared deviations.

LENS 4

Coherence

The change reads as one mind's work; no contradictory or half-migrated patterns.

No plan? The lenses degrade — and say so

With no plan, the conformance lenses degrade to PR-description / commit-message conformance, and the report must state that explicitly. The review is honest about what ground truth it had.

Module 6 · the heart of the skill

The last-mile methodology

Walk every promised behavior through its full chain, citing evidence at every hop. The canonical chain:

trace each promised behavior, hop by hop
UI event → handler → service → HTTP → route → handler → persistence → response → UI update → secondary effects

The named AI failure modes it hunts

  • Optimistic UI — the interface updates with no real call behind it.
  • Leftover mocks — a stub never swapped for the real implementation.
  • Swallowed errors — a catch that hides the failure.
  • Contract drift — a payload that no longer matches what the consumer accepts (even past the repo's seams, to external CLIs / APIs / SDKs).
  • Unregistered routes — a handler that's never wired in.
  • Two halves, different understandings — front and back built to mismatched assumptions.
The non-negotiable synthesis rule

A last-mile finding without a cited hop-by-hop chain is rejected at synthesis. “Save might not work” doesn't qualify; “the chain breaks at hop 4 — the handler at x:120 calls a route that's never registered” does. Requiring a cited chain forces the reviewer to actually trace rather than gesture.

Static-first, then --browser

The method is static-first; --browser escalates to observation against an already-running dev server, citing real request/response traffic — the strongest evidence the review can produce. It never starts a server, never reads .env for ports, and confirms before mutating.

Know which to reach for

Flags & the finder fleet

FlagWhenWhat it changes
(default)Most reviewsOne fresh agent, four lenses + deterministic pre-pass.
--multi-agentLarge/risky diffsFive-stage pipeline: scripted setup → 8 model-tiered single-lens finders → dedup/decompose → severity-gated adversarial verification → script-assembled synthesis.
--megaRelease gatesSame pipeline, dials at max — all finders/verifiers on the main tier; ~2× wall clock. Confirm cost first.
--browserLive last-mile proofExercises promised behaviors against an already-running dev server; cites observed traffic.
--triageAfter a reviewRuns only triage; the one mode that writes to the plan.
Model-tier discipline — load-bearing

Finders are model-tiered by what each lens catches, and the model is set explicitly on every launch as a pinned full ID (claude-fable-5 / claude-opus-4-8 / latest claude-sonnet-*) — never a bare alias, never Haiku, any tier, any stage. Blocker/Major lenses run on the main tier; diff-scan/conformance/coherence on mid; the combined quality sweep on cheap, capped ≤4. A run that silently fell to Haiku produced zero findings on a diff with two known Criticals. If any agent ran below its tier — discard and relaunch it.

Finders find, verifiers prove

The eight finders overgenerate (every evidenced sev ≥4 candidate, no caps) but are budgeted — read each chunk once, ~25 tool calls, one decisive trace per candidate. Precision is restored downstream by a verification floor at sev ≥5: Minor-and-up candidates get a fresh adversarial verifier briefed to refute them; sev ≤4 ships unverified with finder evidence intact. Every verification outcome that changed anything happened at sev ≥4 — so the floor restores precision exactly where it pays.

Module 8

Triage — the decoupled decision step

Triage is split out so the review never blocks on a human. On the latest open findings:

  • Blockers auto-accept (user informed) → accepted.
  • Major Minor Nit → HITL, finding by finding: fix / defer (→ Deferreds ledger) / reject.
  • Accepted findings become a fix-phase appended to the plan (Goal / Prerequisites / Files / Steps / Acceptance) so /deep-implement can execute it cold. No-plan mode generates a minimal plan stub.
The design rationale

Why deep-code-review is built this way

Eighteen decisions shape the most intricate skill in the series — many calibrated from measured benchmark runs. Expand each for what it does, the rejected alternative, and why.

1Review / triage separation — the run never edits the plan›
What it does

A finished review leaves every finding open and writes to exactly one place — its own 04-Code-Review/ artifacts + the manifest. All routing/acceptance moves to a separate, opt-in --triage step.

Rejected alternative

The pre-split design had the review perform triage inline, routing acceptance into the plan and auto-accepting Blockers during the run.

Why this choice

So the review never blocks on a human and never edits the plan — it can run autonomously as an unattended gate, while HITL triage happens on the user's schedule. Cleanly separates producer (review) from plan-mutating consumer (triage).

2“Findings stay open” — the review writes no verdicts›
What it does

Every finding is written open in all three artifacts, triage outcomes pending. Even Blockers stay open during the run — the review only flags a Blocker in the certificate verdict, it doesn't route it.

Rejected alternative

Auto-accepting Blockers and writing their fix-phase as part of producing the review.

Why this choice

A consistent open-everywhere state makes the review a pure, side-effect-free producer and keeps findings.json a clean snapshot the Deep-Learn distiller can consume without untangling what the run already acted on.

3Independence: fresh agents, never the implementation transcript›
What it does

Fresh agent(s) get the diff + plan + codebase access but never the implementation transcript. Restated in every finder brief.

Rejected alternative

Passing the implementer's reasoning/transcript for cheaper context priming — “an agent that saw the implementation reasoning will rationalize the same gaps.”

Why this choice

The implementer is too close to the code to see what it left out. A reviewer who never absorbed the implementer's mental model is the one positioned to notice the half-wired chain.

4The last-mile lens as the signature concern›
What it does

Functional Completeness (Last-Mile) is one of four default lenses and the skill's signature lens: enumerate promised behaviors, walk each chain hop-by-hop with evidence per hop, verify the complete behavior. Six named AI failure modes are hunted explicitly.

Rejected alternative

The shallow-checkbox approach — ticking “traced it” without producing the trace. “Save might not work” is rejected; only a cited break qualifies.

Why this choice

The last-mile problem is the series' signature failure. Requiring a cited chain forces the reviewer to actually trace rather than gesture — which is what catches the gap.

5Interaction-edge, re-entry, and external-consumer additions›
What it does

Three new bodies of last-mile hunting, each mapped to a real benchmark bug the review had missed: external-consumer contract drift (past the repo's seams), interaction-edge checks (double-submit guard, processing feedback, navigation fallback), and re-entry/idempotency checks (stale artifacts, “a prior run exists” signal, duplicated side effects).

Rejected alternative

Leaving the lens at “trace the happy-path chain” — it missed all five manually-found bugs in the calibration benchmark.

Why this choice

Each addition is calibrated to a specific miss with an in-text exemplar (the button that spawned a fresh run on every click; the abandoned blueprint leaving files on disk; the navigation crash with no fallback route). Two of these three were later shift-left ported upstream.

6Adversarial test-reading + the Coherence divergent-sibling hunt›
What it does

Changed test files are “suspects, not oracles” — a tautological test (one whose assertion encodes the code's current output, not the intended contract) ships the bug with a passing checkmark. And enumerate parallel implementations of an outbound contract field-by-field; a lone divergence in an external-facing payload is a correctness signal, not a style nit.

Rejected alternative

Treating a green test suite as evidence of correctness; filing a lone wire-shape divergence as a style nit.

Why this choice

The benchmark miss it exists for: a vision client serialized an image one way, a new contract test asserted exactly that shape, the suite passed 24/24, and the real external consumer rejected it with a 500. The test had codified the bug and the pre-pass read green.

7The multi-agent finder fan-out (eight single-lens passes)›
What it does

Stage A is eight independent, fresh, read-only, single-lens finder passes (diff-scan, removed-behavior, seam-trace, last-mile, frozen-state-probe, conformance, coherence, and a combined quality sweep), launched as one parallel batch capped at 8 concurrent. Finders overgenerate above the nit floor but are budgeted.

Rejected alternative

One broad combined agent (“a single agent's lenses anchor on each other; eight fresh agents can't”); earlier 10- then 7-finder rosters; letting finders build verification-grade proofs (“paying for it twice was the single largest cost in the first baseline”).

Why this choice

Calibrated from two measured baseline runs: many narrow independent passes out-find one broad pass, and the cheap-find / expensive-verify split keeps cost proportional. The rematch covered 20 of 24 matrix rows, beating the prior best of 15.

8Severity model + the sev ≥5 verification floor›
What it does

Canonical 1–10 numeric severity mapped to four tiers (9–10 Blocker · 7–8 Major · 5–6 Minor · 1–4 Nit), justified by counting independent user-reachable trigger paths. Under --multi-agent, sev ≥5 candidates get an adversarial verifier; sev ≤4 ships unverified. A finder proposing exactly sev 5 must say why it isn't a 4.

Rejected alternative

Verifying everything including the nit tail (consumed ~¼ of the run for zero changed verdicts); lowering the floor mid-run; words-as-canonical (the number is canonical, words derived).

Why this choice

Every verification outcome that changed anything happened at sev ≥4 — the floor restores precision where it pays and stops spend where it didn't. Trigger-path counting gives severity an objective, checkable basis.

9Model-tier resolution and the Haiku ban›
What it does

Three tiers (main/mid/cheap) resolve from the orchestrator's own model at launch, set explicitly on every launch as a pinned full ID. Blocker/Major lenses on main; diff-scan/conformance/coherence on mid; quality on cheap (capped ≤4). Haiku is never used — any tier, any stage.

Rejected alternative

Bare aliases (a run with the opus alias got Opus 4.6, silently below tier); implicit/inherited resolution (“exactly how the Haiku mis-route happened”); Haiku anywhere (“worse than no output — it ships false confidence”).

Why this choice

A run whose main-tier lenses silently fell to Haiku produced 0 Blockers and 0 Majors on a diff with two known Criticals. Tiering is calibrated by what each lens catches. Any agent shown running below tier is invalid — discard and relaunch.

10Stage 0 scripted setup + script-assembled synthesis›
What it does

Everything deterministic is scripted: diff pre-chunking with an index (finders never ingest the monolithic diff), the pre-pass launched in parallel with finders, verifier fact pre-harvest, situational matching by grep. Synthesis assembles findings.json, rollups, funnel line, nit table, refuted appendix, and certificate scaffold from fixed-format verdicts — “the model writes only the judgment.”

Rejected alternative

Each finder re-deriving the chunk split (“the single largest token sink”); running the pre-pass serially (~13 min idle for nothing); hand-writing the whole report (~103k tokens of transcription; a 161 KB report); per-dimension reports (“one aggregated, deduped, verified report is the contract”).

Why this choice

Determinism trades no accuracy — deterministic work belongs in scripts where it's cheaper and exact, freeing the model's tokens for the judgment only it can make.

11Deterministic pre-pass — tools, not agents›
What it does

Lint, typecheck, tests run as deterministic tools with conventional severities. Commands discovered from the host (package.json → CLAUDE.md → tool configs), never .env. Always run the affected suite as one combined invocation; attribute failures by a scripted base-vs-head two-worktree diff.

Rejected alternative

Linting as an LLM dimension; an agent reasoning out which failures are pre-existing (~13 min, four-plus suite runs for what one comparison classifies exactly); per-file-only test runs (can report green over a suite that fails on a fresh checkout).

Why this choice

Compilers and linters are faster, cheaper, and exact; offloading frees review agents to hunt what tools can't see. The combined-run and base-vs-head rules each close a specific real miss.

12seam-trace and frozen-state-probe specialists›
What it does

seam-trace: for every value the diff writes, who reads it — and when the reader is external, does the serialized shape match what that consumer expects? Establish ground truth (sibling impl → documented schema → else file an unverified-external-contract finding). frozen-state-probe: for every persisted value whose writer/clearer/repair the diff deletes, run surviving readers against the concrete frozen value. “No probed chain, no finding.”

Rejected alternative

Reading the reader's code forward only — “both longest-standing benchmark misses were exactly this shape… the code looked fine read forward; it failed only at the frozen value.”

Why this choice

frozen-state-probe adds what the other dimensions don't: the reader's behavior at the concrete value. It was added specifically to land two rows that survived every benchmark run. Overlap is deliberate; synthesis dedupes.

13Situational checks — progressive disclosure›
What it does

A sparse catalog is read every run; a check's full body loads only when its Match pattern hits the diff, then augments an existing brief — never adds agents. Fired checks are disclosed in the report header. Triggers err toward loading (a false match costs one read; a false miss costs the bug class).

Rejected alternative

Baking every surface-specific hunt into the always-on lenses (dead weight on 90% of diffs); pre-reading bodies to gauge relevance (“the Match pattern is the relevance test”); porting the mechanism upstream (deliberately not — only one proven check exists).

Why this choice

Mirrors how the skill system manages context: pay only for the few checks that fire. It scales to dozens of checks at near-zero standing cost.

14Scope resolution — ask on any ambiguity›
What it does

Three input modes (no-arg, PR, explicit paths). Base detection: PR-declared base → merge-base comparison → ask on ANY ambiguity. The resolved scope is printed as one verbatim line before any launch and reproduced into every artifact.

Rejected alternative

Guessing a base branch (“Do not pick ‘probably main’”); disturbing the user's checkout (prefer gh pr diff + read-only at PR head).

Why this choice

“Wrong scope poisons everything downstream”; the scope line is the contract for the whole review — everything downstream cites it.

15Browser verification — observed, never inferred›
What it does

--browser escalates last-mile from tracing to observation against an already-running dev server, citing observed request/response. Three hard rules: never start a server, never read .env for ports, confirm before mutating. Static-first; browser is the escalation when tracing leaves real doubt.

Rejected alternative

Starting a dev server if none is running (“never” — it collides with the user's processes/ports); inferred claims as evidence (“the network log line that isn't there is the evidence”).

Why this choice

Observed traffic is the strongest evidence the review can produce — where the named last-mile failure modes die or get caught. Static-first keeps the common case cheap.

16Machine-readable findings.json, certificate, report›
What it does

Three artifacts: report.md (human triage doc), findings.json (machine record — numeric severity canonical, tier derived, _doc-keys omitted in real files), certificate.md (pass = no Blockers). Shape kept compatible with a thin future in-app importer.

Rejected alternative

A separate annotation file; a words-first severity model.

Why this choice

The machine-readable file is the data source for Deep-Learn — every findings.json feeds the loop. Numeric severity and a stable shape let the distiller (and a future bridge) consume it with no remapping.

17No card loader — code-review is a producer, not a consumer›
What it does

deep-code-review has no scripts/ directory — it's the one built skill that does not load directive cards at runtime.

Rejected alternative

Giving it a loader stanza like its siblings (implicitly rejected by its absence).

Why this choice

In the hub-and-spoke Deep-Learn model, code-review is the producer of lessons (its findings feed the distiller → cards) while upstream skills are the consumers that apply them. A reviewer that applied prevention cards would also blur its role as the independent measurement point of the loop. (Recovered by inference from the design; no explicit prohibition sentence in the skill's own files.)

18The --security seam — documented but inert›
What it does

--security is a documented seam, inert until /deep-security exists. The contract for when it lands: its dimension agents join this fan-out under the same finder rules and finding shape; findings merge into this report's Security section for a single triage pass.

Rejected alternative

A separate security report / separate triage pass (“Merge, don't fork”); implementing the flag now.

Why this choice

Documenting the integration contract up front means the future skill plugs into a fixed shape rather than forcing a redesign. The seam is honest about being inert so a user invoking it today isn't misled.

Watch out

Common mistakes

Launching before stating scope — or guessing a base branch. State it; ask on ambiguity.

Passing the implementation transcript — fresh eyes only.

Reviewing the happy path and calling it complete — that's exactly the last-mile trap.

A finding without a cited chain/evidence — rejected at synthesis.

Letting the review triage — it never decides fix/defer/reject or edits the plan; that's --triage.

Reading .env for commands/ports — never. Use package.json / angular.json / CLAUDE.md, else ask.

Relaxing token discipline mid-run because the diff “feels important” — escalating to --mega is the user's call, before launch.

A bare model alias (or Haiku) in the multi-agent pipeline — pin the full ID per launch.

Prove it

Mastery checklist

  • Stated resolved scope in one line before any agent; asked on ambiguity.
  • Ran the deterministic pre-pass as tools, not agents; never touched .env.
  • Reviewed all four lenses; traced each promised behavior hop-by-hop with evidence.
  • Produced report.md + findings.json + certificate.md with all findings open, nothing written outside 04-Code-Review/.
  • Chose the right flag for the diff's risk and respected the model-tier discipline.
  • Ran --triage as a separate step and routed accepted fixes into a cold-executable fix-phase.
