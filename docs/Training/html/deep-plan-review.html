deep-plan-review

/deep-plan-review independently reviews a finished plan and surfaces where it misaligns — with the user (taste, preferences, recorded decisions) or the codebase (best-practices, patterns, duplicate behavior, conflicts) — then records findings back into the plan. Review only — may edit the plan doc, never source.

The frame

Mental model

The planner and the user are too close to the plan to see its blind spots. So the review runs in fresh agent(s) that receive the plan + codebase access + the user's recorded preferences — but not the planning conversation.

Fresh eyes, codebase-aware, unanchored by how the plan was argued into being. That independence is the value; passing the transcript would destroy it.

Why this is the middle skill

It replaces the slow manual review that normally follows planning — and it's the cheapest place in the whole pipeline to catch a misaligned plan, before a single line of code is written against it.

How to use it

The review, step by step

  1. Module 1

    Locate the plan

    Resolve in order: explicit path → the plan referenced in conversation → the most recent 01-Plan/plan.md. Confirm if ambiguous. Read it in full.

  2. Module 2

    Choose mode

    Single-agent (default): one agent covers both lenses. --multi-agent: fan out one fresh agent per dimension in parallel, then synthesize. Recommend it when the plan is large or multi-phase.

  3. Module 3

    The two review lenses

    Hunt on exactly two lenses: User alignment (taste, preferences, recorded decisions) and Codebase alignment (best-practices, patterns, duplicate/overlapping behavior, conflicts). Details below.

  4. Module 4

    Dispatch the review (briefing discipline)

    Brief each fresh agent with the plan text, the two lenses, read access to the codebase, and pointers to preference sources (MEMORY.md + feedback_* memories, root CLAUDE.md, per-app CLAUDE.md). Do not pass the planning transcript. Single-agent uses one Agent; multi-agent launches dimension agents in parallel, then a synthesis pass.

  5. Module 5

    Present findings

    Group by severity — Blocker Major Minor Nit — each tagged with its lens, with evidence and a concrete recommendation. Lead with Blockers and Majors. Be honest: say plainly when a lens is clean.

  6. Module 6

    Record the review

    Write the full report to 02-Plan-Review/review.md. Record a findings summary + link in the plan. With the user's go-ahead, apply accepted fixes to the plan and route deferrals into the Deferreds ledger. Update the manifest.

  7. Module 7 & 8

    Optional /columbo, then hand off

    /columbo is a fresh-agent completeness gut-check — “could a fresh agent execute this?” When the user is satisfied, the plan is ready for /deep-implement.

Scaling the fan-out

A small plan may warrant only two agents; a large multi-phase plan warrants all four. Don't spawn agents with nothing to chew on. Parallelism is capped at 2–4 per the standing rule against mass-spawning heavy agents — and review agents are the safe case (read-only, no worktree isolation needed because nothing here writes source).

Module 3, expanded

A plan can be wrong in two independent ways

LENS A

User alignment

Does the plan match the user's taste, preferences, and recorded decisions? Four dimensions: choice/preference, taste/style, knowledge fit, coherence with prior decisions. Source of truth: memory / feedback_* / CLAUDE.md.

LENS B

Codebase alignment

Does it match best-practices and existing patterns? Does it duplicate or overlap existing behavior? Does it conflict with what's there? Six dimensions, including behavioral completeness. Source of truth: the repo itself.

Why split them

The two failure modes require different sources of truth — the user's recorded preferences vs the repo. Separating them keeps each finding grounded in its own evidence. A canonical Lens-B catch: the plan rebuilds an existing pickPort() when a PORT_SAFETY_RULE already exists — duplicate, drift-prone logic, caught by name and path.

The findings contract

Severity model + mandatory evidence

SeverityMeans
BlockerThe plan will produce a wrong, conflicting, or duplicate result if executed as written. Must fix before /deep-implement.
MajorSignificant misalignment or risk that should be addressed.
MinorWorth fixing, but won't break execution.
NitStylistic / preference-level.

Every finding carries: Severity, Lens, Title, Evidence, Impact, Recommendation. Severity is anchored to impact on execution, not author confidence.

Two banned behaviors

Vague advisory findings (“consider improving”) — every finding must cite a path, a plan section, or a named existing function. Manufacturing findings to look thorough — if a lens is clean, say so. Both erode trust in the report.

The design rationale

Why deep-plan-review is built this way

Twelve decisions shape this skill. Expand each for what it does, the rejected alternative, and why.

1Rename deep-review → deep-plan-review›
What it does

The skill is named deep-plan-review with slash command /deep-plan-review.

Rejected alternative

The prior name deep-review.

Why this choice

The series now has a second reviewer — deep-code-review — so a bare “deep-review” is ambiguous about what it reviews. “plan” names the artifact under review and avoids collision. The rename was cosmetically safe: consumer wiring references the plugin, not the skill.

2Independence through fresh agents (withhold the transcript)›
What it does

Review runs in fresh agent(s) that get the plan + codebase + recorded preferences — but not the planning conversation. Restated as a hard dispatch rule and repeated for every dimension agent.

Rejected alternative

Letting the planner (or the agent holding the transcript) review its own plan.

Why this choice

A reviewer who read the argument inherits the plan's framing and rationalizations. Withholding the transcript forces judgment against ground truth — the codebase + preferences — not the plan's own narrative. In parallel mode, “four agents can't anchor on each other the way one agent's single pass can.”

3Review-only guardrail — edits the plan doc, never source›
What it does

“Review only — never implement, never edit source.” The only writable surface is the plan document (findings, accepted fixes, deferrals).

Rejected alternative

A reviewer that also applies the code fixes it finds — routed to /deep-implement instead.

Why this choice

Clean separation of concerns: the review can't silently mutate source and blur the line between critique and execution. The plan stays the single source of truth implement executes against.

4Two review lenses (user / codebase)›
What it does

Every pass examines the plan through Lens A (user alignment, 4 dimensions) and Lens B (codebase alignment, 6 dimensions).

Rejected alternative

A single undifferentiated “is this plan good?” pass.

Why this choice

A plan fails in two ways needing two sources of truth (memory/CLAUDE.md vs the repo). Separating them keeps each finding grounded in its own evidence.

5Single-agent default, multi-agent escalation›
What it does

Default single-agent; use parallel only with the flag, but recommend it for large/multi-phase plans.

Rejected alternative

Always-multi-agent (rejected for cost); always-single (rejected for large plans where one pass loses breadth).

Why this choice

Most plans don't justify four agents; one fresh agent already provides the core independence benefit. Multi-agent “trades tokens for breadth and independence” — reserved for plans big enough to pay for it.

6Multi-agent fan-out + synthesis pass›
What it does

Launch dimension agents in parallel (one message, multiple Agent calls) — codebase-duplication, patterns-and-practices, user-alignment, coherence-conflicts — then a synthesis pass dedupes, resolves conflicts, ranks, and presents one clean list. Raw per-agent output stays behind the curtain.

Rejected alternative

Showing the user raw per-agent outputs; sequential dispatch.

Why this choice

Parallel single-dimension agents maximize independence and depth. The synthesis pass is the counterweight — parallel agents raise the same issue twice and may disagree, so a final pass collapses duplicates and takes a position, sparing the human four overlapping reports.

7Agent-type and parallelism caps (Explore vs general-purpose, ≤4)›
What it does

Explore for read-only/codebase-heavy sweeps; general-purpose when an agent must reason across lenses. Parallelism capped at 2–4.

Rejected alternative

Unbounded fan-out — rejected per the standing guidance against spawning many heavy agents.

Why this choice

Agent type is matched to task; the cap honors a standing user guideline. Review agents are the safe case — read-only, so worktree isolation is unnecessary.

8Findings format — severity + mandatory evidence›
What it does

Four-level severity anchored to impact on execution; required fields including Evidence and Recommendation; presentation grouped by severity then lens, ending in a per-lens verdict.

Rejected alternative

Vague advisory findings (banned); manufacturing findings to look thorough (banned — say the plan is clean instead).

Why this choice

Severity tied to what breaks at implement time lets the human triage by execution risk; mandatory evidence makes each finding independently verifiable; the “say it's clean” rule prevents inflation and padding.

9Review/apply separation — findings stay open›
What it does

The review records findings first; applying them is a separate, user-gated step. Each finding carries a Status defaulting to open: open / accepted→fixed in plan / deferred→see Deferreds / dismissed by user.

Rejected alternative

Auto-applying findings as they're found.

Why this choice

Reaches the same review-then-route separation as deep-code-review but by a lighter mechanism — there's no --triage flag here, just the “with the user's go-ahead” gate plus the Status ledger. Dismissed findings stay listed so the decision is on record.

10DLC-001 contract check injected at dispatch›
What it does

At dispatch, each fresh agent is briefed to confirm the State/Data-Flow Contract is present, every row resolved, and every “single source of truth” AC names exactly one source. The check lives as data in DLC-001, loaded at runtime — not in SKILL prose.

Rejected alternative

Hard-coding the contract check into SKILL.md (“Improvements are data, never SKILL.md edits”).

Why this choice

deep-plan owns the contract; deep-plan-review is the check spoke — an unresolved non-Display-only row becomes a blocking finding. Encoding it as a toggleable card keeps the improvement human-gated and removable.

11Behavioral-completeness dimension — CONFIRMS what deep-plan REQUIRES›
What it does

Lens B dimension #6: for user-triggered actions and stateful flows, a missing specification of interaction-edge and re-entry requirements is a finding (no double-submit guard, no processing feedback, no navigation fallback, no re-entry story). A plan with no such surface is clean — say so.

Rejected alternative

Leaving interaction-edge/re-entry to be caught at code-review time — too late, too expensive.

Why this choice

The “shift-left” port as a paired gate: deep-plan REQUIRES the spec, deep-plan-review CONFIRMS it. Trigger-gated so pure refactors pay nothing. The reviewer is the backstop so the requirement can't be silently skipped by the planner.

12Directive-card loader + byte-identical shared files›
What it does

Runs load-active-cards.sh deep-plan-review and treats each directive as a hard requirement. Self-locating loader, exact-token phase matching. load-active-cards.sh and artifact-structure.md are byte-identical copies across every deep-* skill.

Rejected alternative

Cross-skill imports of shared files (rejected by the standalone rule); baking learned improvements into each SKILL.md.

Why this choice

Skills install independently, so each must self-contain — verbatim copies plus a consistency sweep, not a shared import. Exact-token matching prevents the deep-plan/deep-plan-review prefix collision.

Watch out

Common mistakes

Passing the planning transcript — kills the independence that makes the review work.

Inventing findings to look thorough. If a lens is clean, say so.

Vague findings. Every finding cites a path, a plan section, or a named existing function.

Severity inflation — a stylistic nit is not a Blocker; severity must reflect impact.

Editing source. This skill touches only the plan document.

Forgetting to fold the summary into the plan — the full report alone leaves the plan blind to its own review.

Prove it

Mastery checklist

  • Briefed reviewers with plan + lenses + preference sources, never the transcript.
  • Produced findings on both lenses, each with evidence and a concrete recommendation.
  • Grouped by severity, led with Blockers/Majors, called clean lenses honestly.
  • Wrote the full report and the in-plan summary + link.
  • Applied accepted fixes to the plan; routed deferrals to the ledger; updated the manifest.
