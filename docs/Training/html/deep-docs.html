deep-docs

/deep-docs produces context-window-aware documentation of what's been built — a standing, queryable map structured for progressive disclosure, where every claim is anchored to file:line (symbol) and adversarially verified by fresh agents. Documents only — it never edits source.

The frame

Mental model

deep-docs is the series' missing orientation layer. Implement's hand-off notes are change-scoped and discarded; code-review emits defects, not orientation; conventional generators dump one exhaustive human altitude with no notion of "load only this layer."

Your product is a standing, queryable map — built so an agent on a token budget pays only for the slice its task needs.

The signature failure it hunts

The dual failure of AI-facing docs: the stale map (describes intent, not the code as it is) and the bloated dump (must be loaded whole). Anchoring + verify defend against the first; tiering + a tier-0 ceiling defend against the second.

The structure

Three independently-loadable tiers

Each tier loads on its own. A crawling agent reads the always-loaded index, then pulls only the cards and references its task actually touches.

docs/ai-map/ — progressive disclosure
tier-0  MAP.md + index.json   ← always loaded · tiny · hard ceiling (~2,000 est)
tier-1  subsystem cards       ← loaded on touch · one per boundary
tier-2  deep references       ← on demand · earned by real cross-layer flow / non-obvious invariant
Every token figure is an estimate

Cost is a cheap ceil(chars/4) heuristic, labeled an estimate on every surface — ~450 est, ~1.2k est — never presented as exact. Zero dependencies, accurate enough to drive load/skip decisions, and upgradeable to a real tokenizer later (the "estimate" label is the seam).

How to use it

The pipeline, stage by stage

Run the default-mode pipeline end-to-end. Each stage has a spine reference; read it before executing that stage.

  1. Stage 1

    Intake & scope

    Resolve the input mode (effort diff / named subsystem / whole repo), standalone-vs-effort home, and host conventions (source dirs, language, test layout) — never reading .env/secrets. Apply the existing-docs quarantine: index human docs as human-authored · not anchor-verified, never ingest their prose. State the resolved scope in one line before launching anything.

  2. Stage 2

    Survey (fan-out per boundary)

    Discover boundaries directory-primary + heuristic refine (skip vendored/build dirs; split huge dirs by entry-point; merge tiny siblings; cap ~12). Launch one fresh read-only agent per boundary (≤8 concurrent), each blind to the others, to discover entry points, public surface, key types, invariants, and data flows. Every split/merge/cap decision is recorded in coverage.md.

  3. Stage 3

    Three-tier layering with anchors

    Lay the survey output into the three tiers. Every claim carries a path:line (symbol) anchor; the symbol is the nearest preceding declaration. Every emitted file carries the GENERATED — do not hand-edit; regenerate overwrites header.

  4. Stage 4

    Anchor-verify (always-on, fan-out per card)

    Launch one fresh adversarial verifier per card (≤8 concurrent) that re-resolves every anchor by symbol grep and returns accurate | drifted | over-budget. No sampling. A card with any drifted/over-budget anchor blocks publish — the doc-analogue of an unresolved State/Data-Flow contract row.

  5. Stage 5

    Index & coverage

    Emit index.json (concept/symbol → doc + anchor + token-est + load-when + trust), the machine index the crawling agent reads. Hold tier-0 under its ceiling, switching to grouped overflow rather than truncating. Emit coverage.md — the honesty artifact (documented / not-documented + why / card-only / referenced-vs-ingested / capped boundaries) — plus learn-signal.json, the forward-dependent drift feed (Display-only today).

  6. Stage 6

    Place (staging + atomic mv) & report

    Generate into the sibling staging dir docs/.ai-map.staging/ with an .in-progress marker; on success (verify passed) atomically replace docs/ai-map/ via mv. A run that finds an existing marker refuses. Effort mode also writes a 07-Docs/ pointer + the manifest line. Close with the coverage summary and the explicit not-documented list.

The teeth

Anchored or it's fiction

No prose enters a trusted tier without a resolvable path:line (symbol) anchor, and every anchor is adversarially verified. Anchoring is symbol-primary:

  • Verify re-resolves by symbol first, and re-snaps the line number if it drifted (a benign edit above shifted it).
  • It flags drifted only if the symbol is gone — the thing the doc describes no longer exists.
  • Any drifted / over-budget anchor blocks publish for its card.
Why symbol-primary, exhaustive, no sampling

A raw line number is fragile (any edit above shifts it); a content hash is too sensitive (every reformat trips it). Symbol-primary tolerates benign movement while still catching real removal. And because symbol-grep is cheap, exhaustive verification is affordable — the guarantee is binary: if even one anchor is unverified, the whole trusted tier can't be trusted. This mirrors code-review's fresh-agent adversarial verification, re-skinned for docs.

Know which to reach for

Flags & refresh

Every flag is natural-language-first — the plain-language trigger is the primary path (users on Copilot/Codex have no slash-commands or CLI flags); the --flag is a convenience on top.

TriggerSayWhat it does
(default)“map this codebase,” “document what's built”Runs the full default-mode pipeline above.
put under <path>“write the map to docs/foo”Overrides the default docs/ai-map/ output location.
--refresh“refresh the docs,” “regenerate the map”Re-resolves anchors and diffs the file-set against the stored fingerprint, then regenerates only the stale + new subsystems.
Refresh uses a dual git-free signal

--refresh consolidates verify + incremental: (1) re-resolve all anchors → report drift; (2) diff the current source file-set against index.json → catch undocumented additions. A git base ref is an optional accelerator, used only when git is proven reliable for this map; on any dirty/non-git/shallow/brownfield tree the dual git-free signal still drives the decision. A stored alignment_fingerprint drives a convergence model — once anchors stop drifting and the file-set matches across rounds, refresh promotes to the fast git-diff path (and drops back automatically if git stops aligning).

The design rationale

Why deep-docs is built this way

Thirteen decisions shape this skill. Expand each for what it does, the rejected alternative, and why.

1Three independently-loadable tiers (progressive disclosure)›
What it does

Tier-0 MAP.md + index.json (always-loaded, tiny), tier-1 subsystem cards (on touch), tier-2 deep references (on demand) — each loadable on its own.

Rejected alternative

A single exhaustive human-readable dump, the way typedoc/sphinx/docusaurus work — one altitude with no notion of "load only this layer," costing an agent its whole context window to answer a one-subsystem question.

Why this choice

The product is context-window economy: an agent on a token budget should pay only for the slice its task needs. Tiering is the structural defense against the bloated dump half of the signature failure.

2Every claim is anchored or it's fiction›
What it does

No prose enters a trusted tier without a resolvable file:line (symbol) anchor, and every anchor is adversarially verified.

Rejected alternative

Free-prose documentation that describes intent — ordinary doc-generator output. Rejected because it produces the stale map: docs that drift from the code as it actually is.

Why this choice

Anchoring + freshness re-check is the structural defense against the stale map. It's the doc-equivalent of the series' evidence rule (“a path:line, a named symbol, or an observed behavior”) and of an unresolved DLC-001 row blocking a plan.

3Doc location: in-repo, effort-aware, overridable›
What it does

Output defaults to in-repo docs/ai-map/; in effort mode it also writes a 07-Docs/ pointer + manifest line; the path is overridable by argument.

Rejected alternative

A non-version-controlled or out-of-repo store.

Why this choice

In-repo means the map is standing, version-controlled, and diffs with the PR that changed the code — so doc drift is reviewable, not invisible.

4Token cost: the chars/4 estimate›
What it does

Token cost is a cheap ceil(chars/4) heuristic, labeled an estimate everywhere it appears (e.g. ~450 est).

Rejected alternative

A real tokenizer dependency — rejected for the markdown-only / zero-dep baseline.

Why this choice

Zero dependencies fits a markdown-only repo, and the estimate is accurate enough to drive load/skip decisions; it's explicitly upgradeable to a real tokenizer later, with the “estimate” label as the seam.

5Anchoring: symbol-primary with re-snap›
What it does

Anchors are path:line (symbol), symbol-primary: verify re-resolves by symbol first, re-snaps the line number if it drifted (benign), and flags drifted only if the symbol is gone.

Rejected alternative

Content-hash anchoring — rejected as noisy (flags benign edits).

Why this choice

A raw line number is fragile (any edit above shifts it); a content hash is too sensitive (every reformat trips it). Symbol-primary is the right sensitivity for “did the thing this doc describes still exist?”

6Boundary discovery: directory-primary + heuristic refine›
What it does

Subsystem boundaries are discovered directory-primary — split huge dirs by entry-point, merge tiny siblings, cap the count.

Rejected alternative

Import-graph / LSP boundary discovery — rejected as non-portable for the standalone-on-any-repo primary mode; deferred as an opt-in refinement.

Why this choice

Directory structure is universally available; an import graph or LSP is language/tooling-specific and breaks the “runs standalone on any repo” promise. Semantic discovery is kept as a future opt-in layer feeding the same boundary list.

7Anchor-verify: exhaustive, no sampling›
What it does

Always-on, fan-out-per-card verification: one fresh adversarial verifier per card re-resolves every anchor and returns accurate | drifted | over-budget; bounded ~8 concurrency; any drifted/over-budget anchor blocks publish.

Rejected alternative

Sampled/budgeted verification — rejected because an unverified anchor violates the core anchoring guarantee.

Why this choice

Symbol-grep is cheap, so exhaustive verification is affordable, and the guarantee is binary: if even one anchor is unverified, the trusted tier can't be trusted. This mirrors code-review's fresh-agent adversarial verification.

8Tier-0 ceiling: grouped overflow, never truncate›
What it does

Tier-0 has a fixed hard ceiling (~2,000 est); when subsystems exceed it, MAP.md groups them by top-level area with per-group subtotals + load-when triggers, and detail spills into the grouped index — never silently truncated.

Rejected alternative

Silent truncation; or scaling the ceiling with repo size.

Why this choice

A scaling ceiling defeats the purpose (tier-0 must stay tiny and always-loadable); silent truncation reads as “covered everything” when it didn't. Grouping keeps tier-0 under budget while preserving an honest, navigable path to the spilled detail.

9Refresh: one flag, dual git-free signal + convergence›
What it does

A single --refresh: (1) re-resolve all anchors → report drift; (2) diff the current source file-set against index.json → catch undocumented additions; regenerate only stale + new subsystems. Git is an optional accelerator; a stored alignment fingerprint drives a convergence model.

Rejected alternative

Git-only or anchor-only refresh signals.

Why this choice

Git-only breaks on non-git/dirty/shallow trees; anchor-only misses new undocumented files. The dual signal catches both drift and additions without depending on git, while still using git as a fast path when it's reliable — two failure modes need two probes.

10Artifacts are 100% machine-owned›
What it does

docs/ai-map/ is 100% machine-owned; every file carries a GENERATED — do not hand-edit; regenerate overwrites header, and regen is a wholesale overwrite, no merge.

Rejected alternative

A two-rendering output (an agent map and a separate human site) — rejected, markdown is already human-readable, a second rendering just drifts; and merge-on-regen instead of overwrite.

Why this choice

Docs here are an index, not a narrative, so regenerate-whole is correct and merge would invite drift; humans annotate elsewhere. A single machine-owned rendering can't fall out of sync with itself.

11Existing docs: quarantine, not launder›
What it does

Useful existing docs (README/design notes) are indexed in index.json as external pointers marked human-authored · not anchor-verified so the crawler can find them, but their prose is never absorbed into a trusted tier; coverage.md lists referenced-vs-ingested.

Rejected alternative

Ingesting existing-doc prose into the trusted tiers (laundering it) — rejected as a risk in its own right.

Why this choice

Unverified human prose absorbed into a trusted tier silently breaks the anchoring guarantee — the reader can no longer tell anchored fact from inherited claim. Quarantine keeps the trust boundary visible.

12Build by mirroring siblings; rationale stays in docs/Training/›
What it does

The skill is built by mirroring its two closest siblings — deep-code-review (fan-out survey, fresh-agent adversarial verification, evidence/anchor discipline) and deep-plan (skill scaffold, directive-card stanza, artifact/manifest conventions) — reusing host-agnostic machinery verbatim, including a byte-identical copy of load-active-cards.sh. The rationale write-up lives under docs/Training/, never baked into the skill.

Rejected alternative

Inventing new scaffolding; or shipping a references/design-rationale.md inside the skill — “the skill doesn't read it, so it would only bloat references/.”

Why this choice

The series' standalone rule says each skill self-contains and only directives/ is shared, with host-agnostic files copied byte-identical — reuse means copy, not import. Rationale justifies design to a human; the skill never loads it, so it belongs in training material.

13Cross-assistant portability: natural language is primary›
What it does

Natural-language invocation is the primary path; --flags are a convenience layered on top, and --refresh plus every interaction ships a plain-language trigger from day one (e.g. “refresh the docs”).

Rejected alternative

Slash-command / CLI-flag-only invocation (the convention the four built skills grew up with).

Why this choice

The series must be drivable by users on GitHub Copilot or Codex, who have neither Claude-Code slash-commands nor a CLI-flag convention. deep-docs is built as the reference implementation of this portability principle; retrofitting the older skills is a separate sweep.

A reflexive observation

deep-docs “recreates Claude Code's own Skills architecture” — tier-0 is an always-loaded index, tier-1 cards load on touch, tier-2 refs on demand: exactly the progressive-disclosure shape a SKILL.md + references/ uses. The orientation layer for agents is, structurally, the same idea as the skill format that produced it.

Watch out

Common mistakes

Editing source or other docs. deep-docs owns docs/ai-map/ exclusively and changes no application code.

Writing unanchored prose into a trusted tier — every claim needs a resolvable path:line (symbol).

Laundering existing READMEs into a tier. Index them as external-unverified pointers; never ingest the prose.

Letting tier-0 bloat past its ceiling — switch to grouped overflow, never truncate.

Sampling anchors to save time — one unverified anchor and the whole tier can't be trusted.

Reading .env for host conventions — never. Use package.json / angular.json / CLAUDE.md, else ask.

Half-writing the map — all generation goes through staging + atomic mv; a second run while a marker exists refuses.

Prove it

Mastery checklist

  • Generated a docs/ai-map/ an agent can crawl, loading only the slice its task needs.
  • Kept tier-0 under its ceiling — grouped overflow, never truncation.
  • Every claim anchored to path:line (symbol); every anchor verified before publish.
  • Quarantined existing docs as external-unverified; never laundered their prose.
  • Ran --refresh and regenerated only the stale + new subsystems.
  • Touched no application source — documents, never decides.
