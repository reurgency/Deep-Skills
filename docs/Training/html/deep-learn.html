The problem it solves

Why Deep-Learn exists

The benchmark that seeded the series had a lesson in it: every severity-bearing finding was the same class of bug — a write path whose runtime reader resolved a different source ("execution is not config-aware"). None were in-function logic errors. Code-review caught them — late and expensively.

A one-time fix handles that class. But new classes keep appearing. The goal is a standing mechanism that learns each recurring class once and pushes prevention upstream automatically — with a human gate and a kill-switch, so a bad lesson can't silently degrade the skills.

THE INSIGHT

Cheap kills are upstream

The same bug is cheapest to prevent at plan time, next-cheapest at implement time, most expensive at review time. Deep-Learn moves each lesson as far upstream as it can go.

THE UNIT

Classes, not findings

One Blocker shouldn't rewrite a skill. A recurring class should. The loop accumulates root-cause-tagged classes across runs — raw findings are evidence, not lessons.

THE GUARANTEE

Never silent, never stuck

Nothing is enforced without human approval; anything can be switched off in one command; every card carries where it came from and whether it's working.

The core principle

Improvements are data, not prose

A learning utility must never edit a SKILL.md. Prose edits cause skill bloat, drift (three skills phrasing one lesson three ways), merge conflicts, and — fatally — no clean on/off. So every learned improvement is a structured directive card in a shared registry. Each skill carries one stable stanza: "load the active cards for your phase and apply them." The skills stop changing; the registry changes.

If lessons were prose in the skill…

…the skill grows without bound, three skills drift on one idea, every lesson is a merge conflict, and turning a lesson off means editing (and risking) the skill itself. There is no telemetry and no provenance.

Because lessons are cards…

On/off is a status flip. Provenance (which findings, which review) and effectiveness (did the class recur?) are built in. A skill's prose never changes, so it can't bloat or drift — all variability lives in the registry as data.

Why this matters for you

It adapts to your codebase

The taxonomy and cards are not generic best-practices lifted from a blog. They are distilled from your own reviews — so the system learns the way your code actually breaks, the conventions your team keeps, and the libraries your project leans on.

YOUR FAILURE SHAPES

The classes are yours

Each class is a root-cause pattern that recurred in your reviews — e.g. a persistence surface whose reader and writer disagree. The ledger tags and counts the shapes your codebase actually produces.

YOUR CONVENTIONS

The cards encode your practices

A card like "search for an existing helper before defining a new one" enforces your reuse norms; a data-flow contract enforces your state-management discipline. The directive text reflects how your team already wants to work.

YOUR STACK

Triggers fire on your surfaces

A card's trigger keys off the surfaces in your repo — the stores, config, routes, and library seams you actually use — so a lesson loads only when a change touches the thing it protects.

Per-repo by default — the ledger learns where the code lives

Cards and their provenance live with the plugin registry, and the loop is fed by the reviews you run on this project. Whether a future ledger is shared across all your repos or kept strictly per-repo is an open design question — but the animating idea is the same: the system gets sharper the more you use it on your code, not on someone else's.

How it's wired

The architecture — hub, spine, spokes

A single loop connects the skill that finds lessons (code-review) to the skills that apply them (everything upstream), through a shared vocabulary so nothing drifts.

HUB

Pattern Ledger

Accumulating, not "the last run's issues." Stores root-cause-tagged classes with frequency, severity, exemplar finding IDs, first/last-seen, and which phase(s) can prevent each. Fed by every findings.json.

SPINE

Shared Taxonomy

The category vocabulary — the contract between hub and spokes. Without it each spoke re-derives its own categories and they drift. A class maps many-to-many to spokes.

SPOKES

The upstream skills

For classes they own, each drafts a candidate card using its own leverage — plan adds artifact requirements, plan-review adds checks, implement adds exit invariants + probes. They apply nothing and touch no skill.

STORE

Directives Registry

File-per-card, directory-as-lifecycle-state (active/ · shadow/ · candidate/ · disabled/). Where on/off, provenance, and effectiveness live.

The gates

Card lifecycle — nothing enforced without a human

A card's directory is its lifecycle state; a status: field mirrors it. Cards move only through gates — and the gate into active/ is a required human approval.

the state machine
candidate ──(distiller: class clears threshold)──▶ shadow
  shadow  ── evaluated, NOT enforced — logs "would this have changed the plan?"
  shadow  ──(human gate: approve)──────────────▶ active     ◀── HUMAN GATE (required)
  active  ──(monitor: recurrence drops)────────▶ stays active, verdict = working
  active  ──(monitor: class keeps recurring)───▶ disabled (ineffective)    ◀── REMOVAL
  active  ──(monitor: friction / false pos.)───▶ disabled (false-positive)
  disabled ─ kept with full provenance — never silently deleted
Human gate

No card reaches active/ without explicit approval, recorded in promotion.human_approved_by. Auto-promotion waits until telemetry is trustworthy.

Shadow mode

Candidates can be evaluated without being enforced — they log what they would have done. Default off. It's how a lesson earns trust before it gates anything.

Removal is first-class

Cards can be demoted, not only added — the guard against monotonic bloat. Disabled cards stay with provenance so a decision is never lost.

One command, no skill edits

./toggle.sh DLC-001 off stops applying a card (moves it to disabled/, flips status:); on re-applies it; bare ./toggle.sh lists every card and its state. The move and the status field change together so the two never drift, and git diff shows exactly what flipped. That's the whole on/off mechanism — no config, no prose edit.

What a lesson looks like

Anatomy of a directive card

A card is a Markdown file: YAML frontmatter (structured metadata the loop reasons over) + a Markdown body (the directive an LLM injects). One card can carry a phase-scoped section per owner phase — each skill applies only its own section.

DLC-001.md — frontmatter (abridged)
---
id: DLC-001
category: write-no-reader            # a slug from the shared taxonomy
owner_phases: [deep-plan, deep-plan-review, deep-implement, deep-bugfix]
title: Require a State / Data-Flow Contract
trigger: The plan introduces/removes/changes any persisted field,
         store, localStorage key, config surface, or runtime read.
status: active
provenance:
  findings: [CR-001, CR-002, CR-003, CR-004, CR-005, CR-006, CR-008, CR-009, CR-016]
  reviews:  ["pipeline-consolidation@6879399"]
  added:    "2026-06-13"
promotion:
  human_approved_by: "…"          # the required human gate
  recurrence_threshold_met: true
outcome:
  verdict: unproven               # maintained by the Effectiveness Monitor
---

# body: one section per owner phase — each skill applies only its own
## deep-plan — add the contract artifact (required plan section)
## deep-plan-review — dispatch check (assert every row is resolved)
## deep-implement — validate (exit invariant + contract probe)
## deep-bugfix — prove the contract row is closed
FieldWhat it's for
categoryThe taxonomy slug — a card must not invent a category. This is the hub↔spoke contract.
owner_phasesWhich skills apply the card. Matched by exact token so deep-plan never matches deep-plan-review.
triggerThe change shape that makes the lesson relevant — keyed to surfaces in your repo.
provenanceThe findings and review the lesson came from, plus the date added — the audit trail.
promotionThe gate record: who approved it, whether the recurrence threshold was met.
outcomeEffectiveness telemetry: did the class recur since the card went active? Drives auto-demotion.
At runtime

How each skill applies its cards

Every deep-* skill carries one stable loader stanza. It runs a bundled script and treats what prints as a hard requirement for that run.

the one stanza every skill carries
scripts/load-active-cards.sh deep-plan     # each skill passes its own phase
  1. Lazy & scoped

    Cards reach a skill only when it runs

    Loading happens at the start of a run and only for that phase, so unrelated sessions pay nothing. A card applies to deep-implement only if its owner_phases lists that exact token.

  2. Self-locating

    Works wherever the plugin is installed

    The script finds the shared registry relative to its own path — sidestepping the platform limit that plugin-root variables aren't interpolated in skill prose. The script is copied byte-identically into each skill; only the registry is shared.

  3. Apply, don't edit

    A printed directive is a hard requirement

    The skill applies the section addressed to its phase. "No active cards" ⇒ it proceeds normally. On a host without a reliable shell, the skill applies cards by hand from cards/active/.

  4. Toggle, don't silence

    Never edit a card or skill to turn one off

    If a card is wrong or noisy, toggle.sh <ID> off — the human-gated kill-switch — not a prose edit. Editing to silence a lesson is exactly what the "data, not prose" rule exists to prevent.

Code-review is the producer, not a consumer

The upstream skills (plan, plan-review, implement, bug-fix, docs) load cards. /deep-code-review is the one built skill with no loader — in the hub-and-spoke model it is the producer of lessons (its findings.json feeds the loop) and the independent measurement point. A reviewer that applied prevention cards would blur that role.

The worked first card

DLC-001 — the State / Data-Flow Contract

The seed card, shipped active, is the loop run by hand once. Every severity-bearing finding in the benchmark was one family — a write path whose runtime reader resolves a different source — so this card pushes that kill upstream. It would have surfaced ~100% of that review's severity at plan time. One card, four phases:

deep-plan

Add the contract artifact

A required plan section: one row per piece of state the feature reads or writes, with writer path:line, runtime reader path:line, and a "same source?" column you can't fudge. A row with no reader, or a reader that resolves a different source, blocks plan completion.

deep-plan-review

Dispatch check

Assert the section is present, every row is resolved, and every "single source of truth" claim names exactly one source. An unresolved row that isn't explicitly Display-only + Deferred is a finding that blocks the plan.

deep-implement

Exit invariant + contract probe

For each runtime row, the reader is actually wired to the writer's source — then a write → execute → assert read probe. Typecheck and unit tests pass clean on these bugs; the probe is the net they miss.

deep-bugfix

Prove the row is closed

When the defect is this class, the proof-of-fix chain must terminate in reader resolves writer's source — the final hop cites the reader's resolution site. A proof that stops at "the write happens" or "it compiles" leaves the verdict unproven.

Why the row can't be filled without finding the bug

The writer column forces "where does the save land?"; the reader column forces a runtime trace; "same source?" is a binary. You cannot honestly complete the table without discovering a write/read mismatch — which is exactly the class the benchmark proved most expensive to catch late.

Kept honest

Failure modes & guards

A self-modifying system is only safe if it can be wrong, be caught, and be rolled back. Each risk has a structural guard.

RiskGuard
Overfitting to one PR's quirkPromote a class only after recurrence across ≥2 distinct reviews — or high severity. The primary overfitting guard.
Monotonic bloatRemoval/demotion is first-class; the Monitor prunes ineffective cards. Addition and subtraction.
Conflicting cards (opposite pushes)A conflict/dedup check at promotion time.
Silent degradation of the skillsHuman gate + shadow mode + kill-switch + provenance. Cards are data, never prose.
Auto-application runawayStay human-gated until telemetry is proven; only then graduate to guarded-auto.
Skill bloat / driftEach skill carries one loader stanza; all variability lives in the registry.
Honest about the seams

What's live · what's next

Live now — Phase 1

The shared taxonomy, the directives registry with lifecycle directories, the DLC-001 seed card shipped active, and the loader stanza in all six skills. On/off works by hand today via toggle.sh. Every skill loads its active cards at runtime.

Planned — later phases

The Distiller (/deep-learn): findings.json → Pattern Ledger → candidate cards. The Effectiveness Monitor: recurrence telemetry that drives demote / false-positive flags. Then guarded auto-promotion — only once the telemetry is trustworthy.

Why outcome.verdict reads unproven today

That field is maintained by the Effectiveness Monitor, which is a later phase. Until it exists, humans read the same recurrence signal by hand. The seam is honest about being inert rather than pretending to measure.
