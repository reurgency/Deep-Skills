SKILL 5Remediate

deep-bugfix

/deep-bugfix remediates defects the way /deep-implement builds features — but inverted. It clusters findings by shared cause, re-diagnoses independently of the review's hypothesis, fixes minimally at the confirmed cause, proves the fix with a fresh adversarial agent, and contains the blast radius before committing. Fixes code only — never triages, never edits the plan.

Learning objectives

1

Explain why bug-fixing is the inverse of /deep-implement — no trusted spec, modifies working code, "done" means provably gone.

2

Treat the finding's recommendation as a hypothesis, not a spec, and re-diagnose to a confirmed root cause before touching code.

3

Cluster defects by shared cause so one fix closes a family — and catch the abandoned sibling.

4

Run the apply → proof → containment → commit order and know why nothing unproven ships.

5

Choose --reproduce / --worktree / --learn and the mode, and hold the boundary: bug-fix executes, triage decides.

The frame

Mental model

Bug-fixing is forward construction run backwards. deep-implement executes a trusted spec; bug-fixing has no trusted spec (the cause must be diagnosed first), modifies working code (the highest-risk edit class), and is done only when the bug is provably gone, not when "the step ran."

Its signature failure — the analogue of code-review's last-mile lens — is the claimed fix: a change that silences the symptom without addressing the cause, or fixes one member of a cluster and silently abandons its siblings.

Recommendation ≠ spec

Real findings hedge their own cause ("register the route or correct the URL"). It enters the diagnose brief flagged "hypothesis — do not trust." The agent confirms the cause independently first.

No proof ⇒ not fixed

Containment proves you didn't break working code; it does not prove you fixed the bug. A separate fresh adversarial agent must prove it — the fixer never marks its own homework.

Why a dedicated skill

The inversion — four assumptions flipped

Pointing a fresh agent at findings.json and saying "fix these" yields N symptom patches with no proof and no regression guard. deep-implement is structurally the wrong tool because bug-fixing inverts all four of its assumptions:

deep-implement assumes…deep-bugfix faces…
A trusted spec with known stepsAn unknown spec — the cause must be diagnosed first
Adds new codeModifies working code — the highest-risk edit class
Findings are independent unitsFindings often share one root cause — cluster, don't patch each
"Done" = the step ran & validated"Done" = the bug is provably gone
Clustering is a different altitude than review-time dedup

Review dedup merges duplicate reports of one defect. Cluster grouping is coarser — cause-level, across genuinely-distinct defects. The seed corpus is the proof: PR#65's CR-001…006 survived as six distinct findings that shared one cause ("a write path whose runtime reader resolves a different source"). That is the altitude where the real fix lives — and where the abandoned sibling gets caught.

How to use it

The remediation, module by module

  1. Module 1

    Intake & normalize

    Accept a findings.json, a pasted bug report / stack trace, or a failing test → a uniform defect list. Findings pass through losslessly; raw reports get synthesized BF-001… ids with evidence extracted from the trace. Resolve the effort home; no effort ⇒ ask for a name (default: slugified branch), create it + the manifest.

  2. Module 2

    Scope & round start

    Series mode loads the accepted set — triage already decided; don't re-ask. Only open findings ⇒ an interactive scope pick grouped by tier (scoping is selection, not triage; unselected stay open). Re-entry check first, then write round-N/scope.json — selection + provisional cluster plan — before any cluster work.

  3. Module 3

    Cluster by suspected shared cause

    Group defects whose evidence points at one underlying cause (same subsystem + failure shape + overlapping evidence paths). Clusters are hypotheses — diagnosis splits on divergent causes and merges two on a proven single cause.

  4. Module 4

    Per cluster: diagnose + fix (one fresh agent)

    One fresh agent per cluster — remediation needs continuity: the agent that finds the cause authors the fix. Briefed with finding evidence + repo + the plan (context only), never the implementer's or reviewer's transcripts, the recommendation flagged "hypothesis — do not trust." It diagnoses to a confirmed cause, then fixes minimally at the cause. Attempt cap: 2. Collaborative mode gates here — approve before applying.

  5. Module 5

    Adversarial proof

    A separate fresh agent carries the burden of refuting "fixed." Default tier is the static chain-trace — walk the corrected chain hop-by-hop, a cited path:line per hop; a fix without a traced chain is rejected. It must show every finding in the cluster resolved by the one fix, and returns fixed | unproven | regressed.

  6. Module 6

    Containment (blast radius)

    Extract changed symbols → discover their direct dependents (1 hop; LSP/host index, else grep/import-walk) → run the dependents' tests + a full typecheck. Block only on new failures. Exported/shared symbol ⇒ escalate to 2 hops. Commands discovered from the host — never .env.

  7. Module 7

    Gate / commit — prove before commit

    Order per cluster, never reordered: apply → proof verdict → containment → commit. Collaborative presents fix + proof + containment; autonomous commits on fixed + clean containment (fix(<scope>): <cause> — resolves CR-00X, CR-00Y). Nothing unproven ever ships.

  8. Module 8

    Statuses, round artifacts & hand-off

    On fixed (and only then): flip the cluster's findings accepted → fixed in place, append the cluster's record to fix-summary.md (cause vs the review's hypothesis; under --learn, also root-causes.json). Update the manifest's 06 Bug Fix row and point at re-review. The tight loop: review → triage → bug-fix → re-review.

The non-negotiable sequence

The prove-before-commit order

Per cluster, this order never changes. Two of the three verdicts short-circuit the commit:

one cluster, in order
apply fix → proof verdict → containment → commit

proof = fixed      → containment clean → commit the cluster
proof = unproven   → one more diagnose→fix (cap 2 total), then revert + blocker report
proof = regressed  → revert immediately
Two proofs guard two different things

The proof-of-fix is a positive guarantee — the bug is gone, chain-traced (or red→green under --reproduce). Containment is a negative guarantee — nothing that depended on the old behavior broke. You need both, in that order, and neither is the fixer grading itself: proof runs in a fresh adversarial agent, containment runs as deterministic tools.

In-session steering

/skip-cluster ("skip this cluster") reverts edits and moves on, statuses untouched. /show-proof ("show me the proof") prints the current verdict with its full hop-by-hop chain. /widen ("widen the containment") escalates 1 hop → 2 and re-runs. Every flag is natural-language-first — the plain-language trigger is primary; the --flag is a convenience on top.

Model routing is load-bearing

Every agent's model is resolved from the skill's references/model-map.md and set explicitly per launch — never a bare alias, never a host default. The diagnose/fix and proof stages are tiered by what each must do.

The design rationale

Why deep-bugfix is built this way

Nine decisions define the fifth skill of the series. Expand each for what it does, the rejected alternative, and why.

1A dedicated skill at all — not re-running deep-implement›
What it does

Bug-fixing gets its own skill instead of routing findings back through deep-implement's triage-appended fix-phase.

Rejected alternative

Keep deep-implement as the fix-phase executor, or merely wrap it "more nicely" — explicitly rejected; the bar is doing what deep-implement is structurally incapable of.

Why this choice

deep-implement is forward construction from a trusted spec and gates on new failures in changed files. Bug-fixing inverts all four assumptions — unknown spec, working-code edits, shared causes, provably-gone done-ness. "Fix these" yields N symptom patches with no proof and no regression guard.

2Diagnosis before remediation; the recommendation is a hypothesis›
What it does

Two disciplines anchor the skill: the recommendation is a hypothesis, not a spec (re-diagnose independently, may reject the suggested fix), and no proof ⇒ not fixed.

Rejected alternative

Trusting the review's proposed remedy. Real findings hedge their own cause — CR-001 literally says "register the route or correct the service URL."

Why this choice

The inverse of code-review's independence rule: don't anchor on the proposed remedy. A recommendation written by an agent that was guessing at cause is not a spec you can build against.

3Root-cause clustering — one fix per cause›
What it does

Group distinct findings by shared underlying cause and apply one fix per cause.

Rejected alternative

Fixing findings one-by-one (yields symptom patches); leaning on code-review's review-time dedup.

Why this choice

Review dedup merges duplicate reports of one defect; cluster grouping is coarser — cause-level, across distinct defects. PR#65's CR-001…006 were six distinct findings sharing one cause. That altitude is where the real fix lives — and where an abandoned sibling gets caught.

4Proof-of-fix in two tiers — always a real guarantee›
What it does

Never ships a claimed fix. Default is a static chain-trace proof (walk the root-cause chain hop-by-hop, cited evidence per hop, reusing code-review's last-mile methodology); --reproduce upgrades to a dynamic red→green test. Both are gated by an adversarial proof agent.

Rejected alternative

Accepting a claimed fix on the fixer's own say-so.

Why this choice

Containment proves you didn't break working code; it does not prove you fixed the bug — so a separate positive proof is required, checked by someone other than the author. "The fixer does not mark its own homework."

5Reproduce-before-fix is opt-in, not default›
What it does

A failing-then-passing reproduction test is --reproduce, opt-in.

Rejected alternative

Making red→green mandatory for every fix.

Why this choice

Some bugs can't be reproduced cheaply; the static chain-trace gives a real guarantee at the default tier. When a bug genuinely can't be reproduced, "that is itself a reported finding — never a silent skip." Proof scales with the mode chosen at invocation rather than forcing the most expensive proof on every defect.

6Regression containment is default›
What it does

Default-on: identify changed symbols, find their callers/dependents, re-run their tests (--worktree to isolate). Block only on new failures; escalate to 2 hops for exported/shared symbols.

Rejected alternative

deep-implement's validation, which validates the changed file's imports, not its dependents — insufficient for the modify-working-code edit class.

Why this choice

The danger in bug-fixing is regressing the things that depended on the buggy behavior; that risk lives in the dependents, so containment must look outward from the change, not just inward.

7A tuned independence model — continuity to diagnose, fresh agent to prove›
What it does

Independence is tuned per stage: diagnose + fix is one agent per cluster (the agent that found the cause authors the fix), while proof is a fresh, adversarial agent that tries to refute "fixed" and returns fixed | unproven | regressed.

Rejected alternative

The series' default applied uniformly — split every stage across fresh agents. That would sever the diagnose→fix continuity bug-fixing depends on.

Why this choice

The series enforces independence by withholding transcripts; deep-bugfix keeps that DNA exactly where it pays — the proof stage, so the fixer can't grade itself — but relaxes it where continuity is the whole point. The fix agent still gets evidence + codebase + plan but not the implementer's transcript, and the recommendation arrives flagged "hypothesis — do not trust."

8Boundary: triage still decides; bug-fix only executes the accepted set›
What it does

Triage stays the decision step (fix / defer / reject); deep-bugfix is the specialized executor for the accepted set only — no triage decisions, touches neither deferred nor rejected. It replaces deep-implement as the fix-phase executor and also runs standalone on a raw bug report / trace / failing test.

Rejected alternative

Letting the fixer also decide whether to fix (collapsing triage into bug-fix); keeping deep-implement's fix-phase path as the primary route.

Why this choice

One skill, one job. Triage is a decision; bug-fix is an execution; conflating them re-couples what the pipeline deliberately split. Running standalone mirrors how code-review runs on any PR/branch, not just deep-* efforts.

9Deep-Learn --learn emits a cause-truthed record›
What it does

Under --learn (opt-in), emits root-causes.json — cluster → confirmed cause → taxonomy class → fix → findings resolved — to feed the future Deep-Learn Pattern Ledger.

Rejected alternative

Continuing to feed the Distiller only findings.json, which carries the review's guess at root cause.

Why this choice

Fix-time is where cause becomes ground truth — the confirmed cause (proven by the fix) plus which findings collapsed into one cluster is strictly better signal than review's guess. Opt-in because Deep-Learn isn't built yet, making the reader a forward dependency.

Watch out

Common mistakes

Anchoring on the recommendation instead of re-diagnosing — it enters the brief flagged "hypothesis — do not trust."

Committing a symptom patch — a fix without a traced chain (or red→green test) is rejected at proof.

Fixing one cluster member and silently abandoning its siblings — clustering exists to catch exactly this.

Reordering the gate — proof and containment both come before commit; nothing unproven ships; regressed reverts immediately.

Making a triage decision — fix/defer/reject belongs to /deep-code-review --triage. Scoping open findings is selection, not triage.

Editing the plan or a drive-by refactor — minimal fix at the cause; a gap that needs re-planning goes back to /deep-plan.

Passing implementer/reviewer transcripts to any agent — evidence only, fresh eyes.

Committing on main/develop, or reading .env during host-command discovery.

Prove it

Mastery checklist

  • Loaded active directive cards; wrote round-N/scope.json before any cluster work.
  • Re-diagnosed each cluster to a confirmed cause; treated the recommendation as a hypothesis.
  • Clustered by shared cause; every finding in a cluster resolved by the one fix.
  • Ran apply → proof → containment → commit, in that order; nothing unproven shipped.
  • Flipped statuses accepted → fixed only on a fixed verdict; appended fix-summary records.
  • Made no triage decision, edited no plan; handed off to /deep-code-review for re-review.
