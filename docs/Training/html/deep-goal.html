deep-goal

/deep-goal runs one effort through the whole deep-* relay from a single invocation — plan → review → implement → code-review → bugfix → docs — with a rigor level selecting which stages run and how hard they try. A thin conductor: it sequences and verifies; it never writes plans, source, reviews, or docs itself. Optional add-on — not a seventh bundled skill; the six core skills never require it.

In this add-on

Learning objectives

1

Choose a rigor level deliberately — and preview the resolved pipeline with --dry-run before spending anything.

2

Explain the conductor mental model: thin walker, artifacts-only hand-offs, and the advance test.

3

Locate the interactivity boundary — questions live in the planning window; autonomy starts when plan.md exists.

4

Predict the review→fix loop at each rigor: triage thresholds, the re-review cap, convergence vs halt.

5

Classify blockers as spine (HALT) vs tail (CONTINUE) and know the conductor's action for each.

6

Resume a killed or paused run from pipeline.md, and audit the run report for every decision made alone.

The frame

Mental model

The conductor is thin on purpose. It resolves rigor to a stage list from data (rigor-map.json, repo-overridable), then walks the list: interactive stages run inline (planning at mvp/prod — you are present by definition), autonomous stages each get one fresh subagent briefed with a minimal stage briefing. Nothing travels between stages except artifacts under .deep-skills/<effort>/ — never transcripts.

State lives in 00-Manifest/pipeline.md, updated at every stage boundary. Kill the run anywhere; re-invoking deep-goal resumes from it.

The advance test

After every stage the conductor verifies with its own reads: the expected artifact exists at its canonical path AND the skill flipped its own manifest line. A subagent's closing narrative is not evidence.

The interactivity boundary

All human interaction happens in the inline planning window. From the moment plan.md exists, the run is autonomous: it only notifies (halts, gates, budget pauses, completion) and only waits where you asked (--gate, --budget).

One dial, all data

The rigor dial

One level selects — as data, never prose — which stages run, planning interactivity, review thoroughness, what auto-triage accepts, and the re-review cap. Exact stage lists live in templates/rigor-map.json; a repo override at .deep-skills/rigor-map.json can add or reshape levels (validated at launch, never mid-run).

YOLO

Trust the model

Autonomous plan, build, stop. No review, no docs. Spikes, throwaways, "just show me something."

POC

Fast but sanity-checked

Autonomous plan + columbo, one light review, auto-triage accepts Blockers only (≥ 9) — everything else deferred, on the record. No re-review.

MVP · recommended

You plan, it builds

Interactive planning (1 round), plan review, single-agent code review, triage accepts Blocker + Major (≥ 7), exactly one re-review, docs.

PROD

Full rigor

Deep interactive planning (3 rounds), multi-agent reviews, triage down to Minor (≥ 5), loop must converge (cert PASS + strictly fewer findings each round), cap 3, then docs.

Auto-deferred, never auto-rejected

Below-threshold findings get status deferred + a Deferreds-ledger row and are listed loudly in the run report. Nothing is ever auto-rejected; Blockers (9–10) are always accepted regardless of threshold.

How a run works

Launch to report, module by module

  1. Module 1

    Launch: handshake & resolution

    Re-entry check first (a prior pipeline.md → resume/fresh question) → resolve rigor from the merged map (omitted → one structured question, recommends mvp) → version handshake: deep-skills ≥ 0.2.0 or a polite refusal → settle the effort slug → write the initial pipeline.md. State exists from launch; even a crash during planning is resumable.

  2. Module 2

    Preview with --dry-run

    Say "preview the run": prints the resolved stage list with per-stage modes/flags, triage threshold, re-review cap, gates, and uncalibrated cost bands — then stops. Dispatches nothing, writes nothing, needs no handshake. The cheapest question you can ask the add-on.

  3. Module 3

    The planning window

    Autonomous (yolo/poc): one dispatched deep-plan --autonomous --effort=<slug> (+ columbo per the map) — zero questions; every self-answered decision lands as an Assumptions row, reproduced verbatim in the run report. Interactive (mvp/prod): /deep-plan inline with the map's --rounds. Planning completes when plan.md exists — the autonomy boundary.

  4. Module 4

    Dispatch + advance test

    Per stage: mark in-flight → render the stage briefing (the subagent's entire prompt) → one fresh subagent → advance test → complete the record. One stage in flight at a time. Triage is a second dispatch of deep-code-review, tested on findings.json statuses + the certificate's Triage outcomes table. The conductor is read-only on manifest stage lines.

  5. Module 5

    Boundary machinery

    At every stage boundary, in order: round accounting → budget check → gate check → next dispatch. Pauses land between stages — a dispatched stage always finishes. Notifications fire on exactly four triggers: halts, gates, budget pauses, completion.

  6. Module 6

    Completion

    Write 00-Manifest/run-report.md → update the manifest effort summary (the conductor's only manifest write, ever) → notify → deliver the full report as the closing message.

Where rigor bites

The review→fix loop

Round 1 = code-review → auto-triage at the map's threshold → bugfix. Re-review rounds reuse the level's own code-review entry (same mode, same threshold) and are judged by the round-aware advance test — a pre-dispatch snapshot of findings.json's CR ids / count / reviewed date, so a crash mid-round never marks an un-run round complete.

Cap (from the map)Exit semantics
0 — pocLoop ends after round 1's bugfix; residual deferred findings go to the run report.
1 — mvpExactly one re-review. HALT only on unresolved Blockers (certificate FAIL); otherwise exit to docs with residuals reported.
≥ 2 — prod (3)Converge on certificate PASS + strictly fewer non-fixed findings each round. Cap breached → HALT as a convergence failure.
When a stage says "human needed"

Blocker policy — severity-aware

Classified by where in the spine it happened: upstream blockers poison every later stage; tail blockers are self-contained.

HALT + notify — spine

Plan-review Blocker verdict · implement phase blocked at the fix cap. Downstream would review rubble. The halted pipeline.md is the resume point — re-invoke once unblocked.

CONTINUE + record — tail

An unproven bugfix cluster (clusters are independent) · docs anchor drift (docs-only failure). The run finishes; the report lists them loudly, each with its blocker-report path.

Natural-language-first

Flags

The plain-language trigger is the primary path (Copilot/Codex users have no CLI flags); --flag syntax is a convenience on top.

FlagSayWhat it does
(goal argument)"run deep-goal on <feature>"The feature/goal to take through the pipeline.
--rigor=<level>"at mvp rigor," "yolo it"Select the rigor level. Omitted → one structured question (recommends mvp).
--dry-run"preview the run"Print the resolved pipeline + cost bands, then stop. Dispatches nothing; exempt from the handshake.
--budget=<band>"cap spend at ~500k tokens"Soft token ceiling, checked between stages; crossing pauses + notifies; resume continues.
--gate=<stage>"pause before implement"Human checkpoint before a named stage, even in an autonomous run; persisted; fires once.

The two core hooks it leans on (free-tier features)

deep-plan --autonomous --effort=<slug> [--columbo] [--rounds=<n>] and deep-code-review --triage --auto-accept-min=<sev> shipped in deep-skills 0.2.0 as free-standing features — useful without deep-goal, documented on their own skill pages. That's why the handshake requires ≥ 0.2.0.

Crash-safe by design

Resume & the run report

Re-invoking on an effort with a pipeline.md announces prior state and asks resume / fresh (fresh archives only the pipeline state; artifacts remain). Resume re-enters at the first non-complete record: a stage that finished but wasn't recorded is recognized and skipped (idempotency); one that died mid-flight is re-dispatched once, on the core skill's own internal resume (phase summaries, round scope, --refresh).

The run report — read it end-to-end

At yolo/poc nobody was watching — so everything the pipeline decided alone surfaces in 00-Manifest/run-report.md or nowhere: the plan's Assumptions verbatim, every auto-deferred finding with severity, every blocker with its report path, spend by stage (uncalibrated estimates), and the worktree path + merge instructions when set.

Where it runs

Hosts — Claude Code is the floor

Designed and verified on Claude Code

The per-stage fresh-subagent dispatch assumes Claude Code affordances. On Codex, Cursor, and Copilot deep-goal degrades to inline sequential execution — same stage order, briefings, advance tests, and pipeline.md records, but one shared context window: functional, and prod-rigor runs may hit context limits. The honest per-host matrix, install steps, and the manual per-stage-restart mitigation live in plugins/deep-goal/HOSTS.md.

Watch out

Common mistakes

Treating deep-goal as a seventh bundled skill — it's an optional add-on conductor; the six core skills work identically without it.

yolo/poc on work that matters — autonomous planning means the Assumptions were the model's calls alone. Read them; better, use mvp and be present.

Skipping --dry-run — prod's multi-agent reviews × loop rounds is the expensive corner; preview first.

Expecting mid-run questions — after plan.md exists it won't ask. Want a checkpoint? That's --gate, set at launch.

Trusting a subagent's "done" — the conductor never does; the advance test is the only evidence.

Hand-editing pipeline.md or another skill's manifest line — pipeline state is conductor-owned; stage lines are each skill's own.

Ignoring auto-deferred findings because the run "passed" — deferral is loud by design; triage the report's list like any Deferreds row.

Prod rigor on a non-Claude-Code host — the inline fallback is functional but may hit context limits; see HOSTS.md.

Prove it

Mastery checklist

  • Previewed with --dry-run and chose a rigor deliberately, against the cost bands.
  • Ran an mvp effort end-to-end from one invocation; answered questions only in the planning window.
  • Read a run report and accounted for every Assumption and auto-deferred finding.
  • Correctly predicted HALT vs CONTINUE for a spine and a tail blocker.
  • Killed and resumed a run with no duplicated stage side effects.
  • Authored a repo rigor override that survives launch validation.
