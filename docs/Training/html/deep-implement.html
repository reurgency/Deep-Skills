deep-implement

/deep-implement executes a finished, reviewed plan phase by phase: implement → validate → fix → commit → hand-off — in collaborative or autonomous mode. It is the only deep-* skill that writes source code.

In this skill

Learning objectives

1

Act as an orchestrator: spawn a fresh sub-agent per phase rather than coding yourself.

2

Choose collaborative vs autonomous mode and use --worktree / --parallel correctly — and know when not to.

3

Run the per-phase loop with a bounded fix loop and correct validation gating.

4

Maintain the durable record: Phase Summaries, per-phase commits, hand-offs, Deferreds reconciliation.

5

Recognize the boundary: stop and surface gaps instead of silently re-planning.

The frame

Mental model

You are the orchestrator, not the author. By default you do not write the code — you spawn a fresh sub-agent per phase, validate its work, drive the fix loop, commit checkpoints, and hand off.

You reuse existing machinery (test runner, worktrees, hand-off/resume) rather than reinventing it. The plan is the contract; you execute exactly what it specifies.

Lightweight by design

This is a plan-driven orchestrator — deliberately not a heavy DB-backed pipeline. Keep it slim. Separating orchestration from authoring keeps validation/commit/hand-off discipline outside the agent that has motivated reasons to call its own work done.

The heartbeat

The per-phase loop

for each phase
implement  → fresh sub-agent, briefed with plan + THIS phase + prior summaries
   ↓
validate   → scoped typecheck + tests; block only on NEW failures in changed files
   ↓
fix        → bounded loop, cap: 2 attempts, else blocker report
   ↓
commit     → mode-dependent: autonomous checkpoints; collaborative gates first
   ↓
summarize  → append to Phase Summaries appendix IN THE PLAN
   ↓
hand-off   → slim next-phase hand-off, then continue
How to use it

The run, module by module

  1. Module 1

    Inputs & flags

    Plan: explicit path → plan in context → most recent 01-Plan/plan.md. Read it fully, including Phase Summaries and Deferreds. --autonomous/--collaborative set mode (ask if omitted on a multi-phase plan). --worktree isolates work; default is the current branch. --parallel is opt-in, only for provably independent phases.

  2. Module 2

    Load & assess

    Identify single- vs multi-phase, which phases are done (read the summaries), the ledger, and each phase's acceptance criteria. If resuming, start at the first unfinished phase — this is where resumability pays off.

  3. Module 3

    Worktree / branch setup

    With --worktree, create the isolated worktree. Otherwise branch-first if on main/develop. Code changes go in the worktree; plan/summary updates go to the source branch root.

  4. Module 4

    Execute phases

    Sequential (default): for each phase, spawn one fresh sub-agent briefed with the plan + that phase + prior summaries. It implements only that phase. --parallel (guarded): only for phases verified independent.

  5. Module 5

    Validate each phase

    Typecheck (e.g. bunx tsc from the app dir — never npx from root), run relevant tests via the test-runner, and the prompt-assembly snapshot test if the phase touched that pipeline. Block only on new failures in changed files / their imports.

  6. Module 6

    The bounded fix loop

    If validation fails, the phase agent fixes and re-validates. Cap: 2 attempts. Still failing → stop, write a blocker report, notify, and in autonomous mode do not commit broken code.

  7. Module 7

    Commit / gate (mode-dependent)

    Collaborative: present diff + validation + summary; await approve/request-fix; commit only if the user wants. Autonomous: on green, commit a per-phase checkpoint (conventional message + Co-Authored-By trailer), append the summary, write the hand-off, continue.

  8. Module 8 & 9

    Summarize, hand off, finish

    Append a summary to the plan's Phase Summaries appendix and write a slim next-phase hand-off. At the end, write 03-Implementation/summary.md, check off acceptance criteria, update the manifest, and reconcile the Deferreds ledger. Hand off to /deep-code-review.

Phase Summaries live in the plan

The plan document is the durable record. 03-Implementation/summary.md does not replace the Phase Summaries — those stay in the plan so a fresh agent can resume by reading the appendix, finding the first phase not marked done, and continuing. Worktree copies are ephemeral and lost on removal, which is why plan/summary/hand-off updates go to the source branch root.

The two nets

Validation gating & the fix loop

GATING

New failures only

Validation is scoped to the changed files and their imports. Pre-existing errors elsewhere are expected noise. You block only on new failures. This keeps gating fast and signal-rich on a long multi-phase run.

FIX CAP

Two attempts, then hand back

A fixed small bound converts open-ended thrash into a deterministic hand-back to a human. The blocker report captures what failed, what was tried attempt-by-attempt, the suspected cause, files in flight, and the specific human decision needed.

Green ≠ verified for external shapes (DLC-001 + construction rules)

A clean typecheck and green unit tests are structurally blind to a write whose runtime reader resolves a different source. So DLC-001 adds a contract probe (write → execute → assert read), and two construction rules are appended to the phase-agent brief: verify external shapes against ground truth before coding to them, and don't write tautological contract tests (a test asserting the shape you just authored passes regardless of whether the consumer accepts it).

The design rationale

Why deep-implement is built this way

Thirteen decisions shape this skill. Expand each for what it does, the rejected alternative, and why.

1Orchestrator, not author: fresh sub-agent per phase›
What it does

The main session orchestrates and does not write code itself by default. Each phase agent is briefed with the full plan, its phase only, prior summaries, and the latest hand-off, and is told “Implement only this phase… Do not exceed scope… Do not commit.”

Rejected alternative

The orchestrator coding directly; one long-lived agent across all phases; letting the phase agent own validation/commits/summaries.

Why this choice

“A fresh agent per phase keeps context lean over long runs and matches the plan's ‘any phase executable cold’ design.” Separating orchestration from authoring keeps discipline outside the agent motivated to call its own work done.

2Phase-by-phase execution with checkpoints›
What it does

One phase at a time through implement → validate → fix → (commit) → summarize → hand-off; autonomous produces a per-phase checkpoint commit.

Rejected alternative

Implement the whole plan in one pass; replicate the heavy DB-backed story pipeline (“deliberately not that; don't duplicate that machinery”).

Why this choice

Phased execution with checkpoints makes the run resumable, lean, and recoverable — each commit is a known-green restart point. It keeps the skill a lightweight, plan-driven orchestrator.

3Execution modes: collaborative vs autonomous›
What it does

Collaborative = a review gate between phases. Autonomous = continuous execution; on green commit and continue, on a blocker hard-stop. Ask if mode is omitted on a multi-phase plan; the user may switch modes at any phase boundary.

Rejected alternative

A single fixed cadence; silently defaulting to one mode on multi-phase plans.

Why this choice

The two modes map to the user's presence. Autonomous assumes the user enabled bypass-permissions and walked away; collaborative keeps them in the loop. Switching mid-run lets a user go autonomous until a risky phase, then collaborative.

4Bounded fix loop (cap: 2 attempts)›
What it does

On validation failure, fix and re-validate, capped at 2 attempts; still failing → stop, blocker report, notify, never commit broken code.

Rejected alternative

Unbounded retry until green; committing partial/broken work after exhausting attempts.

Why this choice

A fixed small bound converts open-ended thrash into a deterministic hand-back. (The exact number 2 vs 3 is stated but not separately justified in the sources.)

5The blocker-report mechanism + template›
What it does

When the cap is hit, emit a structured report appended to Phase Summaries that also drives the notification: when/after how many attempts, branch & worktree (uncommitted — broken code never committed), what failed, what was tried, suspected cause, files in flight, the specific human decision needed.

Rejected alternative

Silently skipping a failing phase; a free-form failure note.

Why this choice

A structured stop-and-handback makes the failure actionable without conversation history — it states the decision required and preserves in-flight state.

6Phase Summaries + slim hand-off: the durable spine›
What it does

After each phase, append a summary to the plan's appendix and write a slim hand-off. Resume = read the appendix, find the first phase not done, continue. Summaries live in the plan, not in 03-Implementation/summary.md.

Rejected alternative

Progress in conversation/agent memory; moving per-phase detail into summary.md; reusing /handoff machinery exclusively (kept as an optional add-on).

Why this choice

Fresh-agent resumability: pinning the durable record to the portable plan document (not chat or ephemeral artifacts) is what guarantees a cold resume.

7Validation strategy: scoped, new-failures-only, + the contract probe›
What it does

Scoped to changed files + imports; block only on new failures. Per-app typecheck with bunx from inside the app dir (never npx tsc from root, which fetches a stub that hangs); tests via the test-runner with an explicit SUCCESS marker; a snapshot test trigger; plus the DLC-001 contract probe.

Rejected alternative

Block on the whole repo's state; trust green typecheck + unit tests as proof; run heavy behavioral validation every phase.

Why this choice

Scoping keeps gating fast and signal-rich. The probe exists because a clean typecheck is structurally blind to a write whose runtime reader resolves a different source — the exact bug class Deep-Learn was built to kill upstream.

8External-contract + adversarial-test construction rules›
What it does

When a phase produces a payload an external consumer reads, two rules are appended to the brief: (1) verify external shapes against ground truth before coding to them — “neither repo code nor a repo test is ground truth for an out-of-repo consumer”; (2) don't write tautological contract tests. Restated as “Green ≠ verified for external shapes.”

Rejected alternative

Treating a green in-repo test as confirmation of an external contract; letting an unverified shape pass silently (record it as unverified instead).

Why this choice

An explicit shift-left: porting two code-review hardening principles upstream as preventative gates, each mapped 1:1 to a benchmark bug the review caught only after the fact. These are the failure modes that pass typecheck and tests while the feature is broken.

9Commit & branch policy; worktree documentation rule›
What it does

Never commit on main/develop — branch first as feat/<plan-slug>. Conventional messages ending with the Co-Authored-By trailer; never push unless asked. Code stays in the worktree, but plan/summary/hand-off updates go to the source branch root.

Rejected alternative

Default to a worktree (default is the current branch); commit broken code or commit on a default branch; push automatically.

Why this choice

Per-phase green-only checkpoints give the recoverable spine; branch-first protects shared branches; the worktree-doc split keeps the durable record from being lost when an ephemeral worktree is removed.

10Parallelism is opt-in and safety-gated›
What it does

Sequential by default; --parallel required, and even then must prove independence on all three checks: no shared files, no ordering dependency, contract pre-defined. Canonical safe case: one agent builds frontend, another backend, both coding to a shared contract. Bounded to a small handful, never a swarm.

Rejected alternative

Auto-parallelize to save time (“never auto-parallelize; the user must ask, and the safety checks must pass”); unbounded fan-out.

Why this choice

Parallel phases that share files or ordering produce silent corruption; gating on a pre-defined contract is what makes concurrency safe. The bound respects a standing rule against parallel token burn.

11Sparse notifications›
What it does

Notify only on two events: a blocker (fix cap hit) and autonomous full-run completion. Mechanisms in preference order: PushNotification, a native macOS banner fallback, a user-extensible Notification hook.

Rejected alternative

Notify between successful phases / in collaborative mode; adding a Slack/push hook silently (offer to scaffold instead).

Why this choice

“Over-notifying erodes trust in the signal.” Notifications are reserved for the moments that genuinely need the user — a hard stop or “I'm done” — aligning with autonomous mode's premise that the user stepped away.

12Scope boundary: forward construction only›
What it does

Builds only what the plan specifies; on a gap or conflict mid-execution it stops and surfaces it rather than silently re-planning (“that's /deep-plan's job”). Gates on new failures in changed files, not on dependents or pre-existing behavior.

Rejected alternative

Diagnosing root causes, demanding reproductions, or verifying dependents didn't regress — deliberately out of scope and assigned to a future /deep-bug-fix.

Why this choice

Keeping implement a pure forward-constructor from a trusted spec is what lets it be lean, phased, and resumable. Diagnosis-first remediation under regression risk is a structurally different problem.

13Directive-card loader + byte-identical shared-file rule›
What it does

Runs load-active-cards.sh deep-implement and treats each directive as a hard requirement; cards are toggled, never edited. DLC-001 gives implement a validate role — exit invariant + contract probe.

Rejected alternative

Editing SKILL.md prose to encode a lesson (“fatally — no clean on/off”); cross-skill references to one shared file (rejected by the standalone rule).

Why this choice

Improvements are data, not prose: one stable loader stanza, all variability in the toggleable registry — clean on/off, provenance, and telemetry without mutating the skill.

Watch out

Common mistakes

Writing the code yourself instead of orchestrating sub-agents (the default is fresh agent per phase).

Committing on main/develop — always branch first. Never commit failing code.

Parallelizing without --parallel or without proving independence.

Blocking on pre-existing failures instead of only new failures in changed files.

A third fix attempt — the cap is 2; then stop and report.

Silently re-planning when you hit a gap. Stop and surface it — re-planning is /deep-plan's job.

Over-notifying — notify on blockers and autonomous completion only, not routine progress.

Prove it

Mastery checklist

  • Orchestrated phases via fresh sub-agents; you did not hand-author the code by default.
  • Chose mode deliberately; branched before any commit; never committed broken code.
  • Gated only on new failures in changed files/imports; honored the 2-attempt fix cap.
  • Appended Phase Summaries to the plan and wrote each next-phase hand-off.
  • Finished with 03-Implementation/summary.md, manifest update, and a reconciled Deferreds ledger.
