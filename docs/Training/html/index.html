The one-loop overview

One clean idea → merge loop

The Deep-* series is four skills, each doing exactly one job. They chain into a single workflow — and each one hands a self-contained artifact to the next, so a fresh agent could pick up anywhere.

01Produce

Plan

/deep-plan

A steerable planning session that emits a resumable plan document. Plans only — writes no code.

 02Critique

Plan review

/deep-plan-review

Fresh, codebase-aware agents critique the plan against the user and the repo — never the source.

 03Execute

Implement

/deep-implement

Phase-by-phase orchestration with bounded fix loops and checkpoints. The only skill that writes source.

 04Verify

Code review

/deep-code-review

Multi-agent, evidence-gated review that catches the last mile. Review and triage are split.
Who this is for

Engineers who will use the deep-skills on their own features, and maintainers who will extend them. No prior exposure to the plugin is assumed — you should be comfortable with Claude Code, git, and reading a codebase. Two further skills, deep-bug-fix and deep-docs, are on the roadmap and covered in the bonus episode.

Watch this before any episode

Four ideas that recur everywhere

Internalize these once and every per-skill decision reads as an instance of a few ideas. This is the spine of the whole series.

PRINCIPLE 01

One skill, one job

The pipeline is split so no skill does two things. Plan only plans. Plan-review only critiques. Implement is the only writer of source. Code-review only reviews — and even splits reviewing from triaging. When a skill is tempted to cross its boundary, it stops and points you to the right sibling.

PRINCIPLE 02

Fresh-agent resumability

Every artifact is written so a fresh agent with only the artifact + the repo can continue cold — no conversation history. That is why plans are self-contained, why phases carry their own context, and why reviews never get the upstream transcript. “Could a fresh agent pick this up?” is the acceptance test for almost everything.

PRINCIPLE 03

Independence through fresh eyes

Both review skills run in fresh agents that never see the upstream transcript. The planner and the implementer are too close to their own work to see its blind spots; the reviewer gets the artifact + codebase + your recorded preferences, but not the argument that produced it. Independence is the source of the catch.

PRINCIPLE 04

The last mile + evidence

The signature failure the system hunts: code where the interface looks wired and the happy path appears connected, but the behavior doesn't make it the last mile — optimistic UI with no real call, swallowed errors, half-wired chains, silently dropped scope. Every finding must cite evidence: a path:line, a named symbol, or an observed behavior.

Why this matters for the video series

These four ideas are the recurring beats. Each skill episode is, at heart, a demonstration of how that skill enforces one or more of them. If you're scripting narration, open every episode by naming which principle it is about to make concrete.

Where everything lands

The artifact tree

Deep-skills runs write to the target repo, not to the plugin repo, under a per-effort directory. Knowing this layout makes the whole pipeline legible — and the plan document is the durable spine: it accumulates phase summaries, a Deferreds ledger, review findings, and a triage fix-phase.

.deep-skills/<effort-name>/
# per-effort directory in the TARGET repo
.deep-skills/<effort-name>/
  00-Manifest/        manifest.md   ← stage statuses (any skill creates it on first write)
  01-Plan/            plan.md       ← /deep-plan  (+ the --triage fix-phase append)
  02-Plan-Review/     review.md     ← /deep-plan-review
  03-Implementation/  summary.md    ← /deep-implement
  04-Code-Review/     report.md · findings.json · certificate.md  ← /deep-code-review
  05-Security/        (reserved for a future /deep-security)
The plan is the spine — learn to read it

Co-locating every stage's output under one effort tree gives the durable spine the series depends on. The manifest is a single status table linking each stage's artifact, so a fresh agent can find the whole history of an effort from one place. Per-project (not per-user) placement keeps artifacts with the repo they describe.

The self-improving loop

Deep-Learn — improvements are data, not prose

The skills themselves never change to encode a lesson. Instead, recurring issue classes are distilled into toggleable directive cards — structured data the upstream skills load at runtime via scripts/load-active-cards.sh <phase>.

PRODUCER

code-review finds it

Every /deep-code-review run emits a machine-readable findings.json. Recurring bug classes across runs are the raw material for the loop.

DISTILLER

a card is authored

A class becomes a directive card with clean on/off (toggle.sh <ID> on|off), provenance, and effectiveness telemetry. Human-gated — nothing reaches active/ without approval.

CONSUMERS

upstream skills apply it

plan / plan-review / implement load active cards at runtime and prevent the same bug earlier next time — without anyone editing a skill by hand.

The seed card — DLC-001

The first card adds a State / Data-Flow Contract to plans: one row per piece of state read or written, where “the reader resolves the same source the writer wrote to” is a binary you can't fudge. It would have surfaced ~100% of one benchmark PR's review severity at plan time. You'll meet it again in Episodes 1–3.

Trainee rule

When a skill prints a directive at the start of a run, treat it as a hard requirement for that run. Don't edit cards or skills to silence one — toggle it.

The capstone exercise

Run all four, unassisted

Pick a small, real feature in a repo you know, and run the full loop end-to-end:

  1. /deep-plan it — choose an effort name; produce a resumable plan with a Data-Flow Contract.
  2. /deep-plan-review it in --multi-agent mode; apply accepted fixes to the plan.
  3. /deep-implement it collaboratively, one phase at a time.
  4. /deep-code-review the branch, then /deep-code-review --triage.
You've mastered the system when…

A teammate can open only your .deep-skills/<effort>/ directory — no chat history — and reconstruct what was built, why, what was reviewed, and what remains deferred.
