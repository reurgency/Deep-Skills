deep-plan

/deep-plan runs a guided, interruptible feature-planning session and produces a self-contained, fresh-agent-resumable plan document. Plan only — it never implements.

In this episode

Learning objectives

1

Run a planning session end-to-end and produce a plan.md a fresh agent could execute cold.

2

Choose the right question cadence and keep questions decision-relevant.

3

Drive the in-session commands (/drill, /breakout, /gaps, /risks, /constraints, /columbo) without losing your place.

4

Structure a multi-phase plan and maintain a Deferreds ledger so nothing is lost.

5

Fill the State / Data-Flow Contract (directive DLC-001) so cross-layer bugs surface at plan time.

The frame

Mental model

You are the session host, not a code author. Your product is a document. The user steers at any time with in-session commands; you service the command and return to exactly where you were.

The deep work is asking sharp questions, reusing before inventing, and writing down enough that the conversation becomes disposable.

The single acceptance test

Could a fresh agent with only this plan + the repo succeed? Everything you write is judged against that one question.

How to use it

The session, module by module

The workflow is an eight-step interruptible state machine. Run the modules in order, but watch for steering commands at every step.

  1. Module 1

    Setup & framing

    Restate the feature in one sentence and confirm it. Pick a question cadence. Propose a kebab-case effort name — the plan lands at .deep-skills/<effort>/01-Plan/plan.md. State scope, explicit non-goals, and your assumptions; get a quick confirm.

  2. Module 2

    Explore before you ask (read-only)

    Launch up to 3 Explore agents in parallel to find reusable patterns, integration points, data models, and the files that will be touched. Reuse before inventing — name existing functions with their paths. No edits in this phase. Every question you can answer from exploration is a question you shouldn't ask the user.

  3. Module 3

    Question rounds

    Emit questions per the chosen cadence, each rendered via AskUserQuestion from a JSON template. Keep them decision-relevant — skip anything exploration already answered. Honor /drill and /breakout mid-round without losing place.

  4. Module 4

    In-session commands — the steering wheel

    Watch for steering commands at every step. Service, then resume exactly where you were. (Full table below.)

  5. Module 5

    The pre-write nudge

    Before writing anything, ask plainly: “Want a /gaps, /risks, or /constraints pass before I write the plan?” Honor any, all, or none — repeatably. Only write when the user is ready. This is a mandatory step, never skipped.

  6. Module 6

    Design & write the plan

    Write to 01-Plan/plan.md using the template. Include self-contained Context / Approach / Steps / Files / Verification, reused functions with their paths, a Deferreds ledger, and the State / Data-Flow Contract. If multi-phase, structure each phase to run cold and leave an empty Phase Summaries appendix for /deep-implement. No source edits — only the plan document.

  7. Module 7

    Deferreds — nothing forgotten

    Anything you defer goes in the ledger with three fields: What, Why deferred, Integration (how it connects back when done). Never bury a deferral in prose.

  8. Module 8 & 9

    The contract, then review & hand off

    Fill the State / Data-Flow Contract (below). Present the plan, fold in feedback, update the doc. Offer /columbo last as the final completeness check; fix what it surfaces. That's the hand-off point to /deep-plan-review and /deep-implement.

The most underrated module

Module 1 is the one to slow down on. Produce just the restatement, three non-goals, and an effort name — then stop. It's the most-skipped, highest-value step, and the discipline of stopping early is what the whole skill is about.

Module 4, expanded

The steering wheel

These commands work at every step. The skill recognizes them whether typed as a slash command or as a plain keyword — so they work for users on Copilot or Codex too.

CommandWhat it does
/drill <note>Deepen the current question in this session (shared context, full history), then resume.
/breakout <note>Dispatch an isolated fresh subagent with a compact briefing; return only its distilled answer — protects main context.
/gapsGap-analysis round; surface gaps / conflicts / assumptions as follow-ups. Repeatable.
/risksRisk assessment of the emerging plan.
/constraintsSuggest likely constraints and ask the user for theirs; fold accepted ones in.
/columboFresh-agent “one more thing” completeness gut-check on the written plan.
/drill vs /breakout — a core skill

Drill stays in the shared conversation (cheap, contextual). Breakout spins an isolated agent (protects your main context from a deep tangent). Reach for breakout when the side-quest would otherwise flood the session — the away-team returns only the “pearl,” not the whole transcript.

Module 8 · directive DLC-001

The State / Data-Flow Contract

A required table: one row per piece of state the feature reads or writes at runtime.

#StateWriter(s) path:lineRuntime reader(s) path:lineSame source?Resolution / AC
1………✓ / ✕…

A row is resolved only if the runtime reader resolves the same source the writer wrote to. Otherwise it blocks plan completion unless marked Display-only and moved to Deferreds.

The four rules:

  • Every new persisted store appears as a writer row.
  • Every “single source of truth” AC points to exactly one source.
  • Deletion-parity: removing a surface re-homes each behavior it provided.
  • Reader = none, or reader-source ≠ writer-source ⇒ the row blocks.
Why it has teeth

You can't fill the row honestly without discovering the bug. The writer column forces “where does the save land?”; the reader column forces a runtime trace; “Same?” is a binary you can't fudge. This single card would have surfaced ~100% of one benchmark PR's review severity at plan time, instead of at code-review time.

The design rationale

Why deep-plan is built this way

Thirteen non-obvious decisions are baked into this skill. Expand each to see what it does, the alternative that was rejected, and why this choice was made.

1Planning-only boundary›
What it does

The skill is hard-bounded to planning: “Plan only — never implement,” restated as a guardrail and as explicit “No code edits” lines. A table assigns one job per sibling skill.

Rejected alternative

The monolithic “plan-and-build” assistant that does everything in one session. deep-plan stops after the plan (+ optional /columbo) and hands off.

Why this choice

Separation of concerns plus resumability: the plan is a durable hand-off artifact, so the skill that produces it must not also mutate the codebase out from under that artifact. Asked to build, it points you to /deep-implement rather than silently crossing the boundary.

2Plan as a durable spine, per-effort tree›
What it does

All artifacts live under .deep-skills/<effort>/, one subdir per effort, with numbered stage folders. deep-plan writes 01-Plan/plan.md and, on first write, creates the manifest if absent. The effort name is a slug deep-plan proposes and the user confirms.

Rejected alternative

A flat per-user plans directory (~/.claude/plans/) — kept reachable only via explicit path. Making deep-plan the sole manifest creator — rejected so a mid-series entry can self-bootstrap.

Why this choice

Co-locating every stage's output under one tree gives the durable spine; the manifest links each stage so a fresh agent finds the whole history from one place. Per-project placement keeps artifacts with the repo they describe.

3Interruptible state machine + always-watch commands›
What it does

An eight-step interruptible state machine; at every step the host watches for commands and, after servicing one, returns “to exactly where you were.” Commands are accepted even without the leading slash if the user clearly means it.

Rejected alternative

A rigid fixed-script interview that runs to completion before the user can intervene.

Why this choice

Steerability is a first-class goal. Accepting keyword forms without the slash ties to cross-assistant portability — slash syntax can't be assumed on every host.

4/drill vs /breakout — in-room dive vs isolated agent›
What it does

/drill deepens the current question inside the session with shared context. /breakout dispatches an isolated fresh subagent with a compact briefing — not the whole transcript — and returns only the distilled pearl, choosing agent type by need.

Rejected alternative

A single “investigate deeper” command. Passing the full transcript to the subagent — explicitly rejected (“NOT the whole transcript”).

Why this choice

Context-budget management: dives that would bloat the main session are pushed into a throwaway agent that returns only its conclusion, while dives that benefit from full shared history stay in-room.

5Question cadence as a user-chosen knob›
What it does

Step 1 asks the user to pick a cadence — 1-at-a-time, 3-at-a-time, or all-at-once — held for the session but changeable any time. All-at-once is the default for small, well-scoped work. Cadence governs delivery, not volume.

Rejected alternative

A single fixed interview rhythm. 1-at-a-time is reserved for cases where each answer reshapes the next question.

Why this choice

Different features have different question-dependency structures, and users have different tolerance for back-and-forth. The “delivery, not volume” rule keeps the knob from becoming an excuse to over-ask.

6Question formats as editable JSON templates›
What it does

Three question types ship as fill-in JSON schemas — multiple-choice, yes-no-explain, open-ended — that the skill fills and renders via AskUserQuestion. Fields are modeled on an existing app type (BlueprintQuestion), reusing a known schema. _* keys are docs-only and stripped before rendering.

Rejected alternative

Hardcoding question shapes in SKILL prose. Instead they're external, clonable files with an “Adding a new type” recipe.

Why this choice

Treating formats as data lets a user tweak the question surface without editing the skill — the same data-not-prose philosophy used for directive cards. Reusing the app schema keeps templates compatible with existing tooling.

7The mandatory pre-write nudge, in natural language›
What it does

A non-skippable prompt before writing offers gaps/risks/constraints passes — any, all, or none, repeatably — followed by a one-line recommendation only “if pertinent.” Phrased in plain language so it works on Copilot/Codex too.

Rejected alternative

Silently writing the plan once questions end. Always recommending a pass — rejected as a reflexive nag.

Why this choice

It forces a deliberate quality gate before the artifact is committed, while keeping the user in control. The natural-language phrasing is again cross-assistant portability.

8Cold-executable phase structuring + empty Phase Summaries appendix›
What it does

Every phase carries Goal, Prerequisites, Files (reuse-before-new), Steps, and Acceptance. “No phase may rely on conversation memory.” A Phase Summaries appendix is reserved empty for /deep-implement to append to.

Rejected alternative

deep-plan pre-writing the per-phase hand-offs (owned by implement). Leaving phase dependencies implicit — rejected; cross-phase deps must be stated and reconciled against the ledger.

Why this choice

Direct expression of fresh-agent-resumability at phase granularity. The empty appendix preserves the durable spine — execution writes back into the same document deep-plan authored.

9The Deferreds ledger (three fields, never prose)›
What it does

Anything deferred goes into a ledger with exactly three fields — What, Why deferred, and Integration (how it connects back). “Never drop a deferral into prose where it gets lost.”

Rejected alternative

Recording deferrals inline in narrative prose — the named, rejected alternative.

Why this choice

“Nothing gets forgotten by phase 4.” The mandatory Integration field guarantees a deferred item carries what it needs to rejoin completed work, rather than evaporating into text a later agent skims past.

10DLC-001 as a runtime directive, not SKILL prose›
What it does

The Data-Flow Contract requirement ships as the active directive card DLC-001, loaded at runtime — not baked into SKILL.md. deep-plan owns it; deep-plan-review checks it; deep-implement validates it with a contract probe.

Rejected alternative

Trusting prose ACs that merely claim a single source of truth. The card forces every such claim to point to exactly one source and treats two writers feeding one concept as a conflict row.

Why this choice

It's “the first and highest-leverage directive,” hand-authored from a real PR, that would have surfaced ~100% of that review's severity at plan time. Catching cross-layer write-no-reader bugs at plan time is far cheaper than at review time.

11The directive-card loader + byte-identical shared files›
What it does

deep-plan runs load-active-cards.sh deep-plan and treats every printed directive as a hard requirement. The loader is self-locating and matches a card to a phase by exact token (so deep-plan ≠ deep-plan-review). Cards are human-gated — toggle, never edit.

Rejected alternative

Encoding each lesson by editing SKILL.md (“Improvements are data, never SKILL.md edits”). Cross-skill references imports — rejected by the standalone rule. Substring matching — rejected for exact-token.

Why this choice

Data-not-prose lets improvements be toggled by a status flip with provenance and effectiveness tracking, without editing a SKILL.md. Byte-identical copies satisfy the standalone rule while a consistency sweep keeps copies in sync.

12Trigger-gated Interaction & re-entry gate (shift-left)›
What it does

For a feature adding a user-triggered action/submit/navigation or a stateful/resumable flow, the plan must fill an Interaction & re-entry section — double-submit guard, processing feedback, navigation fallback, re-entry story. Omitted for pure refactors.

Rejected alternative

Leaving interaction-edge/re-entry to be caught at code-review time — too late and expensive. (The situational-check mechanism and cross-sibling consistency were deliberately not ported.)

Why this choice

A “shift-left port” of a code-review hardening, mapped 1:1 to a real benchmark bug caught only after the fact. Trigger-gating means pure refactors pay nothing — cost is charged only to features that introduce the surface.

13The Explore-agent read-only exploration step›
What it does

Up to 3 parallel Explore agents find reusable patterns, integration points, data models, and touched files — “Reuse before inventing” — and explicitly edit nothing. Reused functions are named with paths in the plan.

Rejected alternative

Inventing new code by default. Read-only is enforced — exploration stays inside the planning-only boundary.

Why this choice

Reuse-before-invent reduces duplication and grounds the plan in real repo paths a fresh agent can follow; capping at 3 read-only agents bounds cost and preserves the no-edits invariant.

A documented wart

The in-session-commands reference still claims each command is a “thin command file in .claude/commands/.” The skill actually recognizes these as keywords whether or not such a file exists — so the assertion is misleading for hosts (Copilot/Codex) that have no such directory. Flagged in the source rationale, not yet fixed.

Watch out

Common mistakes

Writing code. This skill plans only. If asked to build, point to /deep-implement.

Asking what you could have explored. Burn the Explore budget first.

Burying deferrals in prose. They evaporate. Use the ledger's three fields.

A plan that needs the chat to understand. Fails the fresh-agent test — make it self-contained.

Skipping the Data-Flow Contract on a stateful feature — that's exactly where cross-layer bugs hide.

Writing before the pre-write nudge — you skip the user's chance to ask for gaps/risks.

Prove it

Mastery checklist

  • Produced a plan.md a fresh agent executed without the conversation.
  • Chose cadence deliberately and kept every question decision-relevant.
  • Used /drill vs /breakout for the right reasons.
  • Multi-phase plan with cold-executable phases + an empty Phase Summaries appendix.
  • Every deferral in the ledger; every state row resolved or explicitly Display-only/deferred.
  • Ran /columbo as the last gate before hand-off.
