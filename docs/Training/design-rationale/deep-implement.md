# deep-implement — Design Rationale

> `deep-implement` is the execute stage of the `deep-*` trilogy (`/deep-plan` produces → `/deep-plan-review` critiques → **`/deep-implement` executes**) and the **only** deep-* skill that writes source code. Its single job: take a finished, reviewed plan and turn it into committed code, phase by phase, without re-planning. Its signature concern: a phase that *looks* done — green typecheck, green tests — while the feature is actually broken across a layer or contract boundary the in-repo checks never touch. Rationale below is recovered from `SKILL.md`, the six `references/`, two `templates/`, `scripts/load-active-cards.sh`, git commit `3be1254`, and the roadmap docs `DATA-FLOW-CONTRACT.md`, `DESIGN-OUTLINE.md`, and `DEEP-BUG-FIX-DESIGN.md`.

## 1. Orchestrator, not author: fresh sub-agent per phase
**What it does.** The main session is the **orchestrator** and does not write code itself by default — it spawns one fresh sub-agent per phase, validates, drives the fix loop, commits checkpoints, and hands off (`plugins/deep-skills/skills/deep-implement/SKILL.md:10`). Each phase agent is briefed with the full plan, *its phase only*, the Phase Summaries so far, and the latest hand-off (`plugins/deep-skills/skills/deep-implement/references/phase-execution.md:5`). The briefing prompt is fixed and explicitly tells the agent "Implement only this phase… Do not exceed this phase's scope… Do not commit" (`plugins/deep-skills/skills/deep-implement/references/phase-execution.md:29`).
**Alternatives considered / rejected.** Have the orchestrator code directly (rejected — it never implements: `phase-execution.md:11`). Carry one long-lived agent across all phases (rejected in favor of a *fresh* agent per phase). Let the phase agent own validation/commits/summaries (rejected — "The orchestrator handles validation, commits, summaries, and hand-offs — not the phase agent," `phase-execution.md:31`).
**Why this choice.** "A fresh agent per phase keeps context lean over long multi-phase runs and matches the plan's 'any phase executable cold' design" (`phase-execution.md:11`). Separating orchestration from authoring keeps validation/commit/handoff discipline outside the agent that has motivated reasons to call its own work done.

## 2. Phase-by-phase execution with checkpoints
**What it does.** Execution runs one phase at a time, each through the loop **implement → validate → fix → (commit) → summarize → hand-off** (`plugins/deep-skills/skills/deep-implement/SKILL.md:51`). In autonomous mode each green phase produces a per-phase checkpoint commit (`SKILL.md:61`); summaries and hand-offs are written after every phase (`SKILL.md:63`).
**Alternatives considered / rejected.** Implement the whole plan in one pass — implied alternative, rejected by the entire phased structure. Replicate the heavy DB-backed pipeline (`story-implementer`, `.maudel/pipeline_stages/**`) — explicitly rejected: "deliberately *not* the DB-backed story pipeline… Don't duplicate that machinery" (`SKILL.md:36`).
**Why this choice.** Phased execution with checkpoints is what makes the run resumable, lean, and recoverable: each commit is a known-green restart point, and the per-phase artifacts let a fresh agent resume cold (see §6). It keeps the skill a "lightweight, plan-driven orchestrator" (`SKILL.md:36`) rather than reinventing pipeline infrastructure.

## 3. Execution modes: collaborative vs autonomous
**What it does.** Two modes. **Collaborative** = a review gate between phases: implement → validate → fix, then present diff + validation + summary and wait for approve/request-fix; commit only if the user asks (`plugins/deep-skills/skills/deep-implement/references/execution-modes.md:5`). **Autonomous** = continuous execution: on green, commit a checkpoint and continue automatically; on a blocker, hard-stop (`execution-modes.md:18`). The mode is set by `--autonomous`/`--collaborative`; if omitted on a multi-phase plan, **ask** (`SKILL.md:25`). The user may switch modes at any phase boundary (`execution-modes.md:30`).
**Alternatives considered / rejected.** A single fixed cadence — rejected in favor of two named modes for two real situations (user present vs stepped away). Defaulting silently to one mode on multi-phase plans — rejected: the skill asks (`SKILL.md:25`). For single-phase plans the mode question is collapsed — it runs one implement→validate→fix pass and mode only affects whether it gates before committing (`execution-modes.md:3`).
**Why this choice.** The two modes map to the user's presence. Autonomous is summarized as "uninterrupted on success, hard-stop on failure" (`execution-modes.md:28`) and assumes the user has enabled bypass-permissions and walked away (`execution-modes.md:20`); collaborative keeps the user in the loop at each gate when they're present (`execution-modes.md:7`). Switching mid-run lets a user go autonomous until a risky phase, then collaborative (`execution-modes.md:30`).

## 4. Bounded fix loop (cap: 2 attempts)
**What it does.** If validation fails, the phase agent fixes and re-validates, but the loop is capped at **2 attempts**; still failing → stop, write a blocker report, notify, and (autonomous) do not commit broken code (`plugins/deep-skills/skills/deep-implement/SKILL.md:57`). Both modes share the cap (`execution-modes.md:11`, `execution-modes.md:23`).
**Alternatives considered / rejected.** Unbounded retry until green — rejected; the cap is what "keeps a stuck phase from burning the run" (per the training doc's framing, and structurally implied by the hard cap at `SKILL.md:57`). Committing partial/broken work after exhausting attempts — explicitly rejected: "do **not** commit broken code" (`SKILL.md:57`).
**Why this choice.** A fixed small bound converts an open-ended thrash into a deterministic hand-back to a human. `rationale: the exact number 2 (vs 3) is stated but not separately justified in sources`.

## 5. The blocker-report mechanism + template
**What it does.** When the fix cap is hit, the skill stops and emits a structured blocker report appended to the plan's Phase Summaries appendix; it also drives the notification message (`plugins/deep-skills/skills/deep-implement/templates/blocker-report.md:1`). The template captures: when/after how many attempts, branch & worktree (with "Uncommitted (broken code is never committed)"), what failed (scoped to changed files), what was tried attempt-by-attempt, suspected cause, files in flight, the specific human decision needed to unblock, and whether notification fired (`templates/blocker-report.md:3-12`). A trailing comment reasserts: autonomous mode does not proceed past a blocker (`templates/blocker-report.md:14`).
**Alternatives considered / rejected.** Silently skipping a failing phase and continuing — rejected (autonomous "never proceeds past a blocker," `execution-modes.md:28`). A free-form failure note — rejected in favor of a fixed template that doubles as the notification source and records the *exact* human action required.
**Why this choice.** A structured stop-and-handback makes the failure actionable by a human without conversation history: it states the decision required and preserves the in-flight state, consistent with the resumability design (§6).

## 6. Phase Summaries + slim hand-off: the durable, resumable spine
**What it does.** After each phase the skill appends a summary to the plan's **Phase Summaries** appendix (title+status, files, validation verdict, decisions, what the next phase needs — `plugins/deep-skills/skills/deep-implement/references/commit-and-handoff.md:26`) and writes a slim next-phase hand-off (`SKILL.md:63`). The hand-off template carries state/short-sha, what just landed, contracts to honor, the first concrete action, gotchas, and validation to run (`plugins/deep-skills/skills/deep-implement/templates/phase-handoff.md:5-10`). Resume = read the appendix, find the first phase not marked done, continue — "The plan is the source of truth — no conversation memory required" (`commit-and-handoff.md:48`). Phase Summaries deliberately live **in the plan document**, not in `03-Implementation/summary.md` (`SKILL.md:64`).
**Alternatives considered / rejected.** Keep progress in conversation/agent memory — rejected; the whole point is cold resumability (`commit-and-handoff.md:48`). Move per-phase detail into `03-Implementation/summary.md` — explicitly rejected: "Phase Summaries still live in the plan document — `03-Implementation/summary.md` does not replace them" (`SKILL.md:64`, `commit-and-handoff.md:36`). Reuse `/handoff` machinery exclusively — kept as an *optional* maudel-only addition, with the plan-doc hand-off as the portable default (`commit-and-handoff.md:44`).
**Why this choice.** Fresh-agent resumability is a cross-cutting design tenet of the series: each phase must be executable cold from plan + repo. Pinning the durable record to the plan document (which already travels and is portable) rather than to ephemeral artifacts or chat history is what guarantees a fresh agent can resume. The worktree rule reinforces this: code stays in the worktree but plan/summary/hand-off updates go to the source branch root, because worktree copies are ephemeral and lost on removal (`commit-and-handoff.md:11`).

## 7. Validation strategy: scoped, "new-failures-only," + the DLC-001 contract probe
**What it does.** Validation is scoped to the **changed files and their imports**; pre-existing errors elsewhere are expected noise and the skill blocks only on **new** failures (`plugins/deep-skills/skills/deep-implement/references/validation.md:3`, `SKILL.md:54`). The concrete checks: per-app typecheck with `bunx` from inside the app dir (never `npx tsc` from root, which fetches a placeholder stub that hangs — `validation.md:7`), relevant tests via the `test-runner` `execSync` capture (bun TTY output doesn't survive plain redirection; pass only on an explicit `SUCCESS` marker — `validation.md:28`), and a codebase-aware trigger: if the phase touched `.maudel/pipeline_stages/**`, agent or skill prompt inputs, run the prompt-assembly snapshot test (`validation.md:18`). The DLC-001 directive card adds, at §4 Validate, an **exit invariant** that each runtime Data-Flow-Contract row's reader is wired in code, plus a **contract probe** (write → execute → assert read) (`docs/roadmap/DATA-FLOW-CONTRACT.md:36`).
**Alternatives considered / rejected.** Block on the whole repo's typecheck/test state — rejected; pre-existing failures are "expected noise" (`validation.md:3`). Trust green typecheck + unit tests as proof of correctness — rejected for cross-layer bugs: the roadmap notes the current validation suite "would have caught **none** of CR-001–006… the bug is cross-layer. The contract probe is the missing net" (`DATA-FLOW-CONTRACT.md:36`). Run heavy behavioral/Playwright validation every phase — rejected; that runs "only when the phase's acceptance criteria call for it" (`validation.md:32`).
**Why this choice.** Scoping to new failures in changed files keeps gating fast and signal-rich on a long multi-phase run while still catching what the phase actually broke. The contract probe exists because a clean typecheck and green unit tests are structurally blind to a write whose runtime reader resolves a *different* source — exactly the PR#65 bug class the Deep-Learn loop was built to kill upstream (`DATA-FLOW-CONTRACT.md:36`, `DESIGN-OUTLINE.md:11`).

## 8. External-contract + adversarial-test construction rules (commit 3be1254)
**What it does.** Two construction rules are appended to the phase-agent briefing whenever a phase produces a payload an **external consumer** reads or writes a test asserting a contract/external shape (`plugins/deep-skills/skills/deep-implement/references/phase-execution.md:33`). (1) **Verify external shapes before coding to them** — confirm the shape the consumer *actually accepts* against ground truth (sibling client, schema/docs, real probe); "Never code to an assumed external payload shape… Neither repo code nor a repo test is ground truth for what an out-of-repo consumer accepts" (`phase-execution.md:37`). (2) **Don't write tautological contract tests** — a test asserting the shape you just authored passes regardless of whether the consumer accepts it; validate against ground truth, or flag the assertion unverified (`phase-execution.md:38`). The same principle is restated as a verdict rule: "**Green ≠ verified for external shapes**" (`validation.md:34`).
**Alternatives considered / rejected.** Treat a green in-repo test as confirmation of an external contract — explicitly rejected (`validation.md:34`). Let an unverified shape pass silently — rejected: record it as **unverified** rather than letting green imply confirmation (`phase-execution.md:38`, `validation.md:34`). The commit also documents what was *deliberately not* ported alongside these: the situational-check mechanism (only one proven check existed) and cross-sibling consistency (already covered by "code to the shared contract") — see commit `3be1254` body.
**Why this choice.** Commit `3be1254` is an explicit shift-left: porting two deep-code-review hardening principles upstream "as preventative gates, each mapped 1:1 to a real benchmark bug the review caught only after the fact." These are the two AI failure modes that "pass typecheck and tests while the feature is broken" (`phase-execution.md:35`) — the skill's signature concern. Putting them in the phase-agent briefing makes the construction discipline preventative rather than a late review catch.

## 9. Commit & branch policy; worktree documentation rule
**What it does.** Never commit on `main`/`develop` — branch first as `feat/<plan-slug>` (`plugins/deep-skills/skills/deep-implement/references/commit-and-handoff.md:3`, `SKILL.md:71`). Autonomous commits a checkpoint after a phase passes; collaborative commits only when the user asks; messages are conventional `type(scope): <phase title>` and **must** end with the `Co-Authored-By: Claude Opus 4.8` trailer; never push/force-push unless asked (`commit-and-handoff.md:14-22`). Default is to work in the current branch; `--worktree` isolates via `/create_worktree`. The worktree documentation rule (reused from story-implementer): code changes stay in the worktree, but plan/Phase-Summary/hand-off updates go to the source branch root (`commit-and-handoff.md:11`, `SKILL.md:44`).
**Alternatives considered / rejected.** Default to a worktree — rejected; default is the current branch, worktree is opt-in (`SKILL.md:26`, `commit-and-handoff.md:7`). Commit broken code or commit on a default branch — rejected as hard rules (`SKILL.md:71`). Push automatically — rejected (`commit-and-handoff.md:22`).
**Why this choice.** Per-phase green-only checkpoints give the resumable, recoverable spine (§2/§6). The branch-first rule protects shared branches; the worktree-doc split keeps the durable record (which lives in the plan) from being lost when an ephemeral worktree is removed (`commit-and-handoff.md:11`).

## 10. Parallelism is opt-in and safety-gated
**What it does.** Phases run sequentially by default; `--parallel` is required to run concurrently, and even then the orchestrator must first prove independence on **all three** checks: no shared files, no ordering dependency, contract pre-defined (`plugins/deep-skills/skills/deep-implement/references/phase-execution.md:15`). Canonical safe case: one agent builds frontend, another backend, both coding to a shared contract (`phase-execution.md:21`). If any check fails, fall back to sequential and state why (`phase-execution.md:23`); concurrency is bounded to "a small handful, never a swarm" (`phase-execution.md:25`).
**Alternatives considered / rejected.** Auto-parallelize to save time — explicitly rejected: "Never auto-parallelize to 'save time'; the user must ask, and the safety checks must pass" (`phase-execution.md:23`). Unbounded fan-out — rejected per "the user's standing rule against needless parallel token burn" (`phase-execution.md:25`).
**Why this choice.** Parallel phases that share files or ordering produce silent corruption; gating on a contract pre-defined in the plan is what makes concurrency safe. The bound respects an explicit standing user rule against parallel token burn (`phase-execution.md:25`).

## 11. Sparse notifications
**What it does.** Notify only on two events: a blocker (fix cap hit) and autonomous full-run completion; never on routine per-phase progress (`plugins/deep-skills/skills/deep-implement/references/notifications.md:3`, `SKILL.md:72`). Mechanisms in preference order: the `PushNotification` tool (desktop + phone via Remote Control, one line <200 chars), a native macOS `osascript` banner fallback, and a user-extensible `Notification` hook the skill only needs to *emit* into (`notifications.md:7-17`).
**Alternatives considered / rejected.** Notify between successful phases / in collaborative mode — explicitly rejected (`notifications.md:19`). Adding a Slack/push hook silently — rejected: offer to scaffold via `update-config`, "do not add it silently" (`notifications.md:17`).
**Why this choice.** "Over-notifying erodes trust in the signal" (`notifications.md:3`) — notifications are reserved for the moments that genuinely need the user (a hard stop, or "I'm done"), which also aligns with autonomous mode's premise that the user has stepped away.

## 12. Scope boundary: forward construction only, no diagnosis/regression-hunt
**What it does.** The skill builds only what the plan specifies; on encountering a gap or conflict mid-execution it stops and surfaces it rather than silently re-planning — "that's `/deep-plan`'s job" (`plugins/deep-skills/skills/deep-implement/SKILL.md:70`). Validation gates on **new** failures in changed files/their imports (`SKILL.md:54`), not on dependents or pre-existing behavior.
**Alternatives considered / rejected.** Diagnose root causes, demand reproductions, or verify that callers/dependents didn't regress — deliberately out of scope. The bug-fix design contrasts the two directly: "DeepImplement is **forward construction from a trusted spec**: it executes known steps, adds code, and gates on *new* failures in changed files" and "validates the changed file's imports, not its dependents" (`docs/roadmap/DEEP-BUG-FIX-DESIGN.md:13`, `:35`). Those inverted assumptions (unknown spec, modifying working code, shared root cause, proof-of-fix) are assigned to a separate future skill, DeepBugFix, not retrofitted here (`DEEP-BUG-FIX-DESIGN.md:13`).
**Why this choice.** Keeping implement a pure forward-constructor from a trusted spec is what lets it be lean, phased, and resumable. Diagnosis-first remediation under regression risk is a structurally different problem ("the danger is regressing the three things that depended on the buggy behavior," `DEEP-BUG-FIX-DESIGN.md:16`); folding it in would blur the trilogy's clean producer→critic→executor split.

## 13. Directive-card loader + byte-identical shared-file rule
**What it does.** Before starting, the skill runs `scripts/load-active-cards.sh deep-implement` and treats every printed directive as a hard requirement for the run; cards are learned, human-vetted improvements stored as **data**, never baked into the skill, and are toggled via `directives/toggle.sh <ID> off`, never by editing the card or skill (`plugins/deep-skills/skills/deep-implement/SKILL.md:14-20`). The loader is self-locating — it finds the registry relative to its own path so it works regardless of install location or cwd — and matches cards by exact `owner_phases` token so `deep-plan` does not match `deep-plan-review` (`plugins/deep-skills/skills/deep-implement/scripts/load-active-cards.sh:20`, `:35`). Both `load-active-cards.sh` and `references/artifact-structure.md` are copied **byte-identically** into every deep-* skill (`load-active-cards.sh:9`, `references/artifact-structure.md:3`).
**Alternatives considered / rejected.** Edit SKILL.md prose to encode a learned lesson — explicitly rejected: "A learning utility must **never edit a SKILL.md**. Prose edits cause: skill bloat, drift… and — fatally — no clean on/off" (`docs/roadmap/DESIGN-OUTLINE.md:18`). Cross-skill references to a single shared file — rejected by the series "standalone rule": skills self-contain; only the registry is shared, so the shared files are duplicated and a consistency sweep verifies they stay identical (`artifact-structure.md:3`, `load-active-cards.sh:9`).
**Why this choice.** "Improvements are data, not prose" (`DESIGN-OUTLINE.md:16`): the SKILL.md carries one stable loader stanza while all variability lives in the toggleable registry, giving clean on/off, provenance, and effectiveness telemetry without ever mutating the skill. deep-implement's specific leverage in this loop is exit invariants + validation probes (`DESIGN-OUTLINE.md:83`), realized as the DLC-001 contract probe in §7. `rationale (model-tier/agent-type): not specified in sources beyond the Co-Authored-By trailer naming Claude Opus 4.8`.
