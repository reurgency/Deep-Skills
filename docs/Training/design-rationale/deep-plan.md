# deep-plan — Design Rationale

> Training material. `deep-plan`'s single job is to run an interactive, steerable planning session and emit **one self-contained plan artifact** — and nothing else: it never edits source (that's `deep-implement`) and never critiques the plan (that's `deep-plan-review`). The signature concern it addresses is **fresh-agent resumability**: the plan must let an agent with only the doc + the repo execute cold, with no conversation memory. Rationale below is recovered from `plugins/deep-skills/skills/deep-plan/SKILL.md`, its `references/` and `templates/`, the `scripts/load-active-cards.sh` loader, the roadmap docs (`docs/roadmap/DESIGN-OUTLINE.md`, `docs/roadmap/DATA-FLOW-CONTRACT.md`), and git history (`c52ba7e`, `3be1254`). Where a rationale isn't in the sources, it's marked not recovered.

## 1. Planning-only boundary
**What it does.** The skill is hard-bounded to planning: "Plan only — never implement" (`plugins/deep-skills/skills/deep-plan/SKILL.md:8`), restated as a guardrail (`SKILL.md:90`) and as the explicit "**No code edits.**" / "Do not implement" lines (`SKILL.md:26`, `SKILL.md:77`). A table assigns one job per sibling skill (`SKILL.md:24-28`).
**Alternatives considered / rejected.** The rejected alternative is the monolithic "plan-and-build" assistant that does everything in one session. The shared series context establishes the four-skills-one-job-each split (deep-plan → deep-plan-review → deep-implement → deep-code-review); deep-plan stops "after the plan + optional `/columbo`" (`SKILL.md:26`) and hands off rather than continuing into execution.
**Why this choice.** Separation of concerns plus the resumability principle: the plan is a durable handoff artifact, so the skill that produces it must not also mutate the codebase out from under that artifact. If the user asks to start building, the skill points them to `/deep-implement` instead of silently crossing the boundary (`SKILL.md:90`).

## 2. Plan-as-durable-spine + per-effort tree
**What it does.** All artifacts live under the host project's `.deep-skills/<effort-name>/` directory, one subdir per effort, with numbered stage folders `00-Manifest … 04-Code-Review` (`references/artifact-structure.md:9-18`). deep-plan writes `01-Plan/plan.md` (`SKILL.md:71`, `references/artifact-structure.md:22`) and, on first write, creates `00-Manifest/manifest.md` if absent (`SKILL.md:71`, `references/artifact-structure.md:47-48`). The effort name is a kebab-case slug deep-plan **proposes** and the user **confirms** (`SKILL.md:50`, `references/artifact-structure.md:52-54`).
**Alternatives considered / rejected.** A flat per-user plans directory is named and rejected: pre-retrofit plans under `~/.claude/plans/` are kept reachable only via an explicit path argument (`references/artifact-structure.md:56-58`, `SKILL.md:50`). Making deep-plan the sole manifest creator is also explicitly rejected — manifest creation is "*not* deep-plan-exclusive" so a mid-series entry (e.g. reviewing a branch that never had a plan) can self-bootstrap (`references/artifact-structure.md:47-48`).
**Why this choice.** Co-locating every stage's output under one effort tree gives the durable spine the series depends on: the manifest is a single status table linking each stage's artifact (`references/artifact-structure.md:31-42`), so a fresh agent can find the whole history of an effort from one place. Per-project (not per-user) placement keeps artifacts with the repo they describe.

## 3. Interruptible state machine + in-session commands
**What it does.** The workflow is an eight-step "interruptible state machine" (`SKILL.md:45-83`). At **every** step the host watches for user-typed commands; after servicing one it returns "to exactly where you were" (`SKILL.md:30-32`, `references/in-session-commands.md:3`). The command table (`SKILL.md:34-41`) covers `/drill`, `/breakout`, `/gaps`, `/risks`, `/constraints`, `/columbo`, and the skill accepts them without the leading slash "if a user … clearly means it" (`SKILL.md:43`).
**Alternatives considered / rejected.** The rejected alternative is a rigid fixed-script interview that runs questions to completion before the user can intervene. The "return to exactly where you were" rule and the slash-optional recognition (`SKILL.md:43`) instead make steering free and non-destructive at any point.
**Why this choice.** Steerability is a first-class goal ("The user steers with in-session commands at any time," `SKILL.md:10`). Accepting keyword forms without the slash is tied to cross-assistant portability (per the project memory: natural-language invocation is primary, supporting Copilot/Codex users, not just Claude Code) — slash syntax can't be assumed on every host.

## 4. `/drill` vs `/breakout`
**What it does.** `/drill` deepens the current question **inside the session** with shared context and full history, deliberately spending main-session context (`references/in-session-commands.md:7-16`). `/breakout` instead dispatches an **isolated fresh subagent** with a compact briefing — not the whole transcript — and returns only the distilled "pearl" (`references/in-session-commands.md:19-34`, `SKILL.md:37`). breakout chooses agent type by need: `Explore` for read-only investigation, `general-purpose` if it must reason or prototype, with foreground default and optional background run (`references/in-session-commands.md:28-30`).
**Alternatives considered / rejected.** Having a single "investigate deeper" command is implicitly rejected by splitting it into two with opposite context economics — the file frames them as "in-room discussion" vs "the away-team" (`references/in-session-commands.md:21`). Passing the full transcript to the subagent is explicitly rejected ("NOT the whole transcript," `references/in-session-commands.md:23`).
**Why this choice.** Context budget management: deep dives that would bloat or block the main session are pushed into a throwaway agent that returns only its conclusion, protecting main context (`SKILL.md:37`), while dives that benefit from full shared history stay in-room.

## 5. Question cadence as a user-chosen knob
**What it does.** Step 1 asks the user to pick a cadence — 1-at-a-time, 3-at-a-time, or all-at-once — held for the session but changeable any time (`SKILL.md:49`, `references/question-cadence.md:3-9`). all-at-once is the default recommendation for small, well-scoped work (`SKILL.md:49`, `question-cadence.md:9`). Cadence governs delivery, not volume: skip any question you can answer from exploration (`question-cadence.md:13`).
**Alternatives considered / rejected.** A single fixed interview rhythm is rejected in favor of three modes mapped to feature complexity (`question-cadence.md:5-9`); 1-at-a-time is reserved for cases "where each answer reshapes the next question," batching for more independent decisions.
**Why this choice.** Different features have different question dependency structures, and users have different tolerance for back-and-forth; making cadence explicit and adjustable matches delivery to both. The "delivery, not volume" rule (`question-cadence.md:13`) prevents the cadence knob from becoming an excuse to over-ask — questions stay decision-relevant regardless of mode.

## 6. Question formats as editable JSON templates
**What it does.** Three question types ship as fill-in JSON schemas — `multiple-choice.json`, `yes-no-explain.json`, `open-ended.json` — that the skill fills and renders via `AskUserQuestion` (`templates/questions/_README.md:8-12`, `SKILL.md:60`). Each carries a `questionType` matching its filename and documents its type-specific fields in an `_comment` (`templates/questions/_README.md:14-22`). `_*` keys are documentation only and must be stripped before rendering (`templates/questions/_README.md:31`, `multiple-choice.json:2`).
**Alternatives considered / rejected.** Hardcoding question shapes in the SKILL prose is the rejected alternative; instead they are external, clonable files with an explicit "Adding a new type" recipe (`templates/questions/_README.md:24-29`). The fields are deliberately modeled on an existing app type, `BlueprintQuestion` from `apps/shared/types/blueprint-qa.types.ts` (`templates/questions/_README.md:15`), reusing a known schema rather than inventing one.
**Why this choice.** Treating question formats as data lets a user tweak or extend the question surface without editing the skill — the same data-not-prose philosophy used for directive cards (§10/§11). Reusing the `BlueprintQuestion` field shape keeps the templates compatible with tooling that already understands that schema (rationale for the specific app-type lineage beyond reuse: not recovered from sources).

## 7. Mandatory pre-write nudge, in natural language
**What it does.** Step 5 is a non-skippable prompt before writing: "Always ask this before writing — it is a mandatory step at the end of every plan session, never skipped" (`SKILL.md:62-68`). It offers gaps/risks/constraints passes — any, all, or none, repeatably — followed by a one-line recommendation only "if pertinent" (`SKILL.md:66`). It is phrased in plain natural language, "no slash required, so it works for users on Copilot/Codex too" (`SKILL.md:63`).
**Alternatives considered / rejected.** Silently writing the plan once questions end is rejected (the step is mandatory). Always recommending a pass is also rejected — the recommendation is conditional and omitted "if nothing warrants one" (`SKILL.md:66`), avoiding a reflexive nag.
**Why this choice.** It forces a deliberate quality gate before the artifact is committed, while keeping the user in control (honor any/all/none, repeatably). The explicit natural-language phrasing is again cross-assistant portability — the gate must fire for users whose host has no slash commands (`SKILL.md:63`).

## 8. Cold-executable phase structuring
**What it does.** For multi-phase work, every phase entry must carry Goal, Prerequisites/inputs, Files (with reuse-before-new), Steps, and Acceptance/validation (`references/phase-structuring.md:5-12`). "No phase may rely on conversation memory. If a decision matters, it's written in the plan" (`phase-structuring.md:16`). A **Phase Summaries** appendix is reserved empty at planning time for `/deep-implement` to append to (`phase-structuring.md:17`, `SKILL.md:75`, `templates/plan-template.md:66-68`).
**Alternatives considered / rejected.** deep-plan executing or pre-writing the per-phase hand-off documents is explicitly out of scope and owned by `/deep-implement` (`phase-structuring.md:21-27`) — deep-plan "only leaves phases shaped so that loop can run." Leaving phase dependencies implicit is rejected: cross-phase deps must be stated explicitly and reconciled against the Deferreds ledger (`phase-structuring.md:18`, `phase-structuring.md:19`).
**Why this choice.** Direct expression of the fresh-agent-resumability principle at phase granularity: any phase must be executable cold from the doc + repo alone (`phase-structuring.md:3`). The empty Phase Summaries appendix preserves the durable-spine handoff — execution writes back into the same document deep-plan authored.

## 9. The Deferreds ledger
**What it does.** Anything deferred goes into a **Deferreds** ledger with exactly three fields — What, Why deferred, and **Integration** (how it must connect back when done) — and "Never drop a deferral into prose where it gets lost" (`SKILL.md:85-87`, `templates/plan-template.md:58-64`). Phase structuring requires reconciling each phase against this ledger (`phase-structuring.md:19`).
**Alternatives considered / rejected.** Recording deferrals inline in prose is the named, rejected alternative ("never dropped into prose," `plan-template.md:60`; "Never drop a deferral into prose where it gets lost," `SKILL.md:87`).
**Why this choice.** The section header "nothing gets forgotten by phase 4" (`SKILL.md:85`) states the intent: a structured ledger with a mandatory Integration field guarantees a deferred item carries the information needed to rejoin completed work, rather than evaporating into narrative text a later agent skims past.

## 10. DLC-001 State/Data-Flow Contract
**What it does.** A required plan section listing **one row per piece of state the feature reads or writes**, where a row is *resolved* only if the runtime reader resolves the same source the writer wrote to — otherwise it blocks plan completion unless marked Display-only and moved to Deferreds (`docs/roadmap/DATA-FLOW-CONTRACT.md:6-30`). Its declared insertion points are deep-plan SKILL.md §6 (required artifact section) and the §7 Review gate "asserts every row resolved" (`DATA-FLOW-CONTRACT.md:32-34`). Notably this requirement is **not** baked into `SKILL.md` §6/§7 prose — it ships as the active directive card `DLC-001` (`plugins/deep-skills/directives/cards/active/DLC-001.md`), loaded at runtime (see §11). deep-plan is the card's primary owner; deep-plan-review checks; deep-implement validates (`DATA-FLOW-CONTRACT.md:4`, card `owner_phases` line 4).
**Alternatives considered / rejected.** Trusting prose ACs that merely *claim* a single source of truth is the rejected status quo — the card forces every such claim to point to exactly one source, and treats two writers feeding one concept as a conflict row (`DATA-FLOW-CONTRACT.md:28`). The worked example shows why a narrative AC fails: "The row cannot be filled in honestly without discovering the bug" because the Writer/Reader/Same? columns force a runtime trace and a binary you can't fudge (`DATA-FLOW-CONTRACT.md:23`).
**Why this choice.** It is "the first and highest-leverage directive," hand-authored from PR#65, that "would have surfaced ~100% of that review's severity at **plan** time" (`DATA-FLOW-CONTRACT.md:2`, coverage `DATA-FLOW-CONTRACT.md:38-40`). Catching cross-layer write-no-reader bugs at plan time is far cheaper than at code-review time; the card was shipped active immediately because it was already human-vetted (`docs/roadmap/DESIGN-OUTLINE.md:173`, card promotion block lines 12-16).

## 11. Directive-card loader + shared files
**What it does.** Before starting, deep-plan runs `scripts/load-active-cards.sh deep-plan` and treats every directive it prints as a hard requirement for the run (`SKILL.md:12-20`). The loader is self-locating — it resolves the shared `directives/cards/active` registry relative to its own path, three levels up, so it works regardless of install location or cwd (`scripts/load-active-cards.sh:20-22`). It matches a card to a phase by **exact token** so `deep-plan` does not match `deep-plan-review` (`scripts/load-active-cards.sh:35`, comment line 34). Cards are human-gated: never edit a card or the skill to disable one — toggle via `directives/toggle.sh <ID> off` (`SKILL.md:20`). The loader and `references/artifact-structure.md` are copied byte-identically into every deep-* skill (`scripts/load-active-cards.sh:8-9`, `references/artifact-structure.md:2-3`).
**Alternatives considered / rejected.** Encoding each learned lesson by editing SKILL.md is the explicitly rejected alternative: "Improvements are data, never SKILL.md edits" (commit `c52ba7e` message; `docs/roadmap/DESIGN-OUTLINE.md:5`, `:19`). Cross-skill `references` imports are rejected by the standalone rule — each skill self-contains; only the `directives/` registry is shared (`references/artifact-structure.md:3`, `scripts/load-active-cards.sh:9`). Substring phase matching is rejected in favor of exact-token matching to avoid `deep-plan` bleeding into `deep-plan-review` (`scripts/load-active-cards.sh:35`).
**Why this choice.** Data-not-prose lets improvements be toggled by a status flip with built-in provenance and effectiveness tracking, "without ever editing a SKILL.md" (`DESIGN-OUTLINE.md:5`) — the skills stop changing, the registry changes (`DESIGN-OUTLINE.md:19`). Byte-identical copies satisfy the standalone rule (a skill must work if lifted out alone) while a consistency sweep keeps the copies in sync (`references/artifact-structure.md:3`).

## 12. Trigger-gated Interaction & re-entry gate
**What it does.** When a feature adds a user-triggered action/submit/navigation or a stateful/resumable flow, the plan must fill the template's **Interaction & re-entry** section — double-submit guard, processing feedback, navigation fallback, and re-entry (abandon/restart) story — "rather than leaving them implicit." The section is omitted for pure refactors/migrations with no such surface (`SKILL.md:76`, `templates/plan-template.md:45-52`). Added in commit `3be1254`.
**Alternatives considered / rejected.** The commit message frames this as a "shift-left port of code-review hardening": the concern (issue #5) was previously caught only by `deep-code-review` after the fact; it is now a preventative plan-time REQUIRES (with deep-plan-review CONFIRMing). Two alternatives are explicitly *not* ported: the situational-check mechanism (#4) — deferred because "only one proven check exists and each phase would grep a different artifact; revisit when a second check justifies the shared infra" — and cross-sibling consistency (#3), already covered elsewhere (`3be1254` message).
**Why this choice.** Each ported gate is "mapped 1:1 to a real benchmark bug the review caught only after the fact" (`3be1254` message); catching interaction-edge defects (a button that fires twice and spawns duplicate backend runs; a handler that renders nothing and reads as "does nothing"; an unmatched route hard-crashing) at plan time is cheaper than at review time. The gate is **trigger-gated** so "pure refactors/migrations pay nothing" — the cost is charged only to features that actually introduce the surface, keeping the planning overhead proportional.

## 13. Explore-agent read-only exploration
**What it does.** Step 3 launches up to 3 parallel `Explore` agents to find reusable patterns/utilities, integration points, data models, and the files that will be touched — "**Reuse before inventing**" — and explicitly edits nothing (`SKILL.md:56-57`). Reused functions are named with their paths in the plan (`SKILL.md:73`, `templates/plan-template.md:15`).
**Alternatives considered / rejected.** Inventing new code by default is the rejected alternative ("prefer existing functions over new code, and name them with paths," `SKILL.md:57`). Read-only is enforced — "Do not edit anything in this phase" (`SKILL.md:57`) — keeping exploration inside the planning-only boundary of §1.
**Why this choice.** Reuse-before-invent reduces duplication and grounds the plan in real repo paths a fresh agent can follow; capping at 3 parallel agents and keeping them read-only bounds the cost and preserves the no-code-edits invariant. (Specific reason for the cap of 3: not recovered from sources.)

## Wart: stale `.claude/commands/` references
`references/in-session-commands.md` claims each in-session command is a "thin command file in `.claude/commands/`" (`references/in-session-commands.md:3`) and that `/columbo` is "the existing repo command (`.claude/commands/columbo.md`)" (`references/in-session-commands.md:69`). These `.claude/commands/` references are a documented wart: the skill is designed to recognize these commands as keywords whether or not a slash command file exists ("works whether typed as a slash command or recognized as a keyword," same line 3), and the broader design pushes natural-language invocation for cross-assistant portability (project memory; `SKILL.md:43`, `SKILL.md:63`). Asserting the behavior *lives in* `.claude/commands/` files is therefore misleading for hosts that have no such directory — flagged here, not fixed (the task forbids modifying the skill).
