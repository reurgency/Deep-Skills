# deep-plan-review — Design Rationale

> **Single job:** independently critique a *finished plan* — never the code — surfacing where it misaligns with the user or the codebase, then record findings back into the plan. **Signature concern:** *independence through fresh eyes* — the review must not be contaminated by the argument that produced the plan, and every finding must carry severity + evidence so a fresh agent (and the human) can act on it. Rationale below is recovered from `SKILL.md`, the four `references/`, the report template, `load-active-cards.sh`, commits `c52ba7e` (rename + Deep-Learn Phase 1) and `3be1254` (shift-left port), and the roadmap docs `DATA-FLOW-CONTRACT.md` / `DESIGN-OUTLINE.md`.

## 1. Rename deep-review → deep-plan-review
**What it does.** The skill is named `deep-plan-review`, with display name `DeepPlanReview` and slash command `/deep-plan-review` — `SKILL.md:2`, `SKILL.md:6`. The rename touched the directory, frontmatter `name`, the display heading, and every sibling cross-reference (`c52ba7e`).
**Alternatives considered / rejected.** The prior name `deep-review` (commit `c52ba7e` diff: `-name: deep-review` → `+name: deep-plan-review`; `-# DeepReview` → `+# DeepPlanReview`).
**Why this choice.** The series now contains a *second* reviewer — `deep-code-review` — so a bare "`deep-review`" is ambiguous about *what* it reviews. The trilogy line in `SKILL.md:10` makes the disambiguation explicit: `/deep-plan` (produce) → `/deep-plan-review` (critique the plan) → `/deep-implement` (execute), with `deep-code-review` reviewing *code* afterward. "plan" in the name names the artifact under review and keeps it from colliding with the code reviewer. The commit message also notes the rename is cosmetically safe: "Consumer wiring is unaffected (it references the plugin, not the skill)."

## 2. Independence through fresh agents (withhold the planning transcript)
**What it does.** The review runs in **fresh agent(s)** that receive the plan + codebase read access + the user's recorded preferences — *but not the planning conversation* — `SKILL.md:22-24`. The withholding is restated as a hard dispatch rule: "**Do not** pass the planning transcript" (`SKILL.md:42`), and repeated for every multi-agent dimension agent: "None of them gets the planning transcript" (`references/multi-agent.md:7`).
**Alternatives considered / rejected.** Letting the planner (or the in-session agent that holds the transcript) review its own plan — rejected. `SKILL.md:23`: "The planner and the user are too close to the plan to see its blind spots."
**Why this choice.** A reviewer who has read the argument that produced the plan inherits the plan's framing and rationalizations; it is "anchored by how the plan was argued into being" (`SKILL.md:24`). Withholding the transcript forces the reviewer to judge the plan against ground truth (the codebase + recorded preferences) rather than against the plan's own internal narrative. `multi-agent.md:3` extends the same anti-anchoring logic to parallel mode: "four agents can't anchor on each other the way one agent's single pass can."

## 3. Review-only guardrail — edits the plan doc, never source
**What it does.** "**Review only — never implement, never edit source.** You may edit the *plan document* (findings, accepted fixes, deferrals)." — `SKILL.md:8`. Reinforced in Guardrails: "Never modify source files or run the implementation. If the user wants fixes built, point them to `/deep-implement`." — `SKILL.md:60`.
**Alternatives considered / rejected.** A reviewer that also applies code fixes it finds — rejected; that work is explicitly routed to `/deep-implement`.
**Why this choice.** Clean separation of concerns across the trilogy: each skill owns one artifact. The reviewer's only writable surface is the *plan* (recording findings, accepted fixes, deferrals), so the review can't silently mutate source and blur the boundary between critique and execution. This keeps the plan the single source of truth that `/deep-implement` then executes against.

## 4. Two review lenses (User alignment / Codebase alignment)
**What it does.** Every pass examines the plan through **Lens A — User alignment** and **Lens B — Codebase alignment** (`references/review-dimensions.md:5-23`). Lens A has 4 dimensions (choice/preference, taste/style, knowledge fit, coherence with prior decisions); Lens B has 6 (best-practices, code patterns, duplicate behavior, overlapping behavior, conflicts, behavioral completeness). A third `Coherence` lens is admitted in the findings format (`references/findings-format.md:18`, template `review-report.md:12`).
**Alternatives considered / rejected.** A single undifferentiated "is this plan good?" pass — rejected implicitly; the two lenses split the two distinct failure modes a plan has.
**Why this choice.** A plan can be wrong in two independent ways: it can fail *this user* (contradict a stated preference, reinvent something they already solved — `review-dimensions.md:9-12`) or fail *this codebase* (duplicate existing logic, fight existing contracts — `review-dimensions.md:18-23`). The two require different sources of truth — memory/`feedback_*`/CLAUDE.md for Lens A vs. the repo itself for Lens B (`review-dimensions.md:7,16`) — so separating them keeps each grounded in its own evidence. The worked finding in `findings-format.md:25-28` (plan rebuilds existing `pickPort()` when `PORT_SAFETY_RULE` already exists) shows Lens B catching duplicate/drift-prone logic by name and path.

## 5. Single-agent default, multi-agent escalation
**What it does.** Default to single-agent; use parallel mode only when `--multi-agent` / `/multi-agent` is given, but **recommend** multi-agent when the plan is large or multi-phase — `SKILL.md:38-39`. In single-agent mode one agent covers both lenses (`SKILL.md:44`, `review-dimensions.md:3`).
**Alternatives considered / rejected.** Always-multi-agent — rejected for cost; `multi-agent.md:3` frames parallel mode as a trade: "Trades tokens for breadth and independence." Always-single — rejected for large plans where one agent's single pass loses breadth.
**Why this choice.** Most plans don't justify four agents; one fresh agent already provides the core independence benefit. Multi-agent is reserved for plans big enough that breadth + cross-agent non-anchoring pay for the extra tokens (`multi-agent.md:3`). `multi-agent.md:18` makes the scaling explicit: "a small plan may only warrant two agents … a large multi-phase plan warrants all four. Don't spawn agents with nothing to chew on."

## 6. Multi-agent fan-out + synthesis pass
**What it does.** Multi-agent mode launches the dimension agents **in parallel** (one message, multiple `Agent` calls) — `SKILL.md:45`, `multi-agent.md:6` — mapping each numbered dimension to its own agent: `codebase-duplication`, `patterns-and-practices`, `user-alignment`, `coherence-conflicts` (`multi-agent.md:9-14`). A **synthesis pass** then dedupes, resolves conflicts, ranks by severity/lens, and produces one clean list — the per-agent raw output "stays behind the curtain" (`multi-agent.md:22-27`).
**Alternatives considered / rejected.** Showing the user the raw per-agent outputs — rejected; only the synthesized list is surfaced (`multi-agent.md:27`). Sequential dispatch — rejected in favor of parallel, both for speed and so agents can't anchor on one another (`multi-agent.md:3`).
**Why this choice.** Parallel single-dimension agents maximize independence and depth-per-dimension (a duplication agent does "heavy codebase exploration," `multi-agent.md:11`). The synthesis pass is the necessary counterweight: parallel agents will raise the same issue twice and may disagree, so a final pass collapses duplicates keeping the strongest evidence, takes a position on conflicts ("reuse X" vs "X doesn't fit"), and presents a single ranked list so the human isn't handed four overlapping reports (`multi-agent.md:22-27`).

## 7. Agent-type and parallelism caps (Explore vs general-purpose, ≤4)
**What it does.** Use `Explore` agent type for read-only / codebase-heavy sweeps and `general-purpose` when the agent must reason across lenses — `SKILL.md:44`, `multi-agent.md:16`. Parallelism is capped at **2–4** agents (`multi-agent.md:31`).
**Alternatives considered / rejected.** Unbounded fan-out — rejected. `multi-agent.md:31`: "Per the user's standing guidance, avoid spawning many heavy explore+write agents at once."
**Why this choice.** Agent type is matched to task: `Explore` is the cheap read-only sweeper for codebase-duplication/patterns dimensions; `general-purpose` is reserved for the dimensions that need cross-lens reasoning (user-alignment, coherence). The ≤4 cap honors a standing user guideline against mass-spawning heavy agents; the reference notes review agents are the *safe* case ("read-only review agents … but still cap at four") and that "worktree isolation is unnecessary" precisely because nothing here writes source (`multi-agent.md:31`).

## 8. Findings format — severity model + mandatory evidence
**What it does.** Findings carry a 4-level severity — **Blocker / Major / Minor / Nit** (`references/findings-format.md:7-12`) — and required fields: Severity, Lens, Title, **Evidence**, Impact, **Recommendation** (`findings-format.md:15-21`). Severity is anchored to *impact on execution*: a Blocker is "the plan will produce a wrong, conflicting, or duplicate result if executed as written … must fix before `/deep-implement`" (`findings-format.md:9`). Presentation groups by severity then lens, ending with a per-lens verdict (`findings-format.md:31-32`).
**Alternatives considered / rejected.** Vague advisory findings — explicitly banned: "Findings must cite evidence — a path, a plan section, or a named existing function/pattern. No vague 'consider improving.'" (`SKILL.md:61`). Manufacturing findings to look thorough — banned: "If you can't find a problem in a lens, say the plan is clean … rather than inventing findings." (`SKILL.md:63`, echoed `findings-format.md:32`, `review-dimensions.md:32`).
**Why this choice.** The series' second cross-cutting principle is fresh-agent resumability backed by evidence (`path:line` / named symbol). A severity tied to *what breaks at implement time* lets the human triage by execution risk, not by author confidence; mandatory evidence makes every finding independently verifiable by a fresh agent or the user; and the "say it's clean" rule prevents severity inflation and padding that would erode trust in the report.

## 9. Review/apply separation — findings stay open until the user go-aheads
**What it does.** The review *records* findings first; applying them is a separate, user-gated step. `SKILL.md:53`: "With the user's go-ahead, apply accepted fixes **to the plan document** … Route anything deferred into the plan's existing **Deferreds** ledger." The report template carries a per-finding `Status` field defaulting to `open`, with values `open | accepted → fixed in plan | deferred → see Deferreds | dismissed by user` (`templates/review-report.md:19,31-35`).
**Alternatives considered / rejected.** Auto-applying findings as they're found — rejected; application waits on explicit user go-ahead. (Contrast the sibling `deep-code-review`, which split its review run from a separate `--triage` step so findings stay `open` and routing/acceptance is a distinct phase.)
**Why this choice.** `deep-plan-review` reaches the same *review-then-route* separation as `deep-code-review` but by a lighter mechanism — there is **no `--triage` flag here**; the separation is the in-workflow "with the user's go-ahead" gate plus the `Status` ledger on each finding. The reasons are the same: the reviewer's judgment (find + rank) is kept distinct from the human's decision (accept / defer / dismiss), and dismissed findings stay listed with `Status: dismissed by user` "so the decision is on record" (`review-report.md:34`). Deferrals are pushed into the plan's Deferreds ledger (What / Why deferred / Integration) "so it can't evaporate" (`SKILL.md:53`).

## 10. DLC-001 contract check injected at §3 dispatch
**What it does.** Step 3 *Dispatch the review* briefs each fresh agent and is the documented insertion point for the State/Data-Flow Contract check — `SKILL.md:41-46`. The check itself lives as **data** in directive card DLC-001, not in `SKILL.md` prose: its `deep-plan-review` section says "assert the **State / Data-Flow Contract** section is present, **every row is resolved** (reader resolves the writer's source), and every 'single source of truth' AC names **exactly one** source" (`directives/cards/active/DLC-001.md:50-52`; roadmap `DATA-FLOW-CONTRACT.md:35`). The card is loaded at runtime by `load-active-cards.sh deep-plan-review` (`SKILL.md:17`).
**Alternatives considered / rejected.** Hard-coding the contract check into `SKILL.md` §3 — rejected in favor of the directive-card mechanism: "Improvements are data, never SKILL.md edits" (`DESIGN-OUTLINE.md:5`, commit `c52ba7e`).
**Why this choice.** The PR#65 benchmark found that *every* severity-bearing finding was one bug family — a write path whose runtime reader resolves a *different* source ("execution is not config-aware") — caught late and expensively by code review (`DATA-FLOW-CONTRACT.md` header, `DLC-001.md:25`). The card "pushes the kill upstream." deep-plan is the primary owner (it adds the contract section); **deep-plan-review's role is the *check* spoke** (`DLC-001.md:4` "Owners: … deep-plan-review (check)") — it verifies the contract exists and every row is resolved before the plan can pass, with an unresolved non-Display-only row becoming a blocking finding (`DLC-001.md:52`). Encoding it as a toggleable card keeps the improvement human-gated and removable without editing the skill.

## 11. Behavioral-completeness dimension — CONFIRMS what deep-plan REQUIRES
**What it does.** Lens B dimension **#6 Behavioral completeness** (`references/review-dimensions.md:23`, added in commit `3be1254`): for user-triggered actions and stateful flows the plan adds, treat a *missing specification* of the interaction-edge and re-entry requirements as a finding — no double-submit/in-flight guard, no processing feedback, no navigation fallback, or no re-entry story (stale-artifact reconcile / "prior run exists" signal / idempotent re-run). "A plan with no such surface is clean on this dimension — say so."
**Alternatives considered / rejected.** Leaving interaction-edge/re-entry to be caught at code-review time — rejected as too late and too expensive; commit `3be1254` maps the gate "1:1 to a real benchmark bug the review caught only after the fact." Also deliberately *not* ported here: the situational-check mechanism (#4) and cross-sibling consistency (#3) — see the `3be1254` message.
**Why this choice.** This is the "shift-left" port of a deep-code-review hardening principle into the planning phases as a paired gate: **deep-plan REQUIRES** the interaction-edge + re-entry spec and **deep-plan-review CONFIRMS** it (commit `3be1254`: "Interaction-edge + re-entry (#5) -> deep-plan REQUIRES, deep-plan-review CONFIRMS"). It is trigger-gated to features that add an action/submit/navigation or a stateful/resumable flow "so pure refactors/migrations pay nothing" (`3be1254`). The rationale stated in the dimension itself: these omissions "ship as real bugs (duplicate backend runs on repeat clicks, stale leftovers from an abandoned run); the plan is the cheapest place to require them" (`review-dimensions.md:23`). The reviewer is the confirmation backstop so the requirement can't be silently skipped by the planner.

## 12. Directive-card loader + byte-identical shared files
**What it does.** Before reviewing, the skill runs `scripts/load-active-cards.sh deep-plan-review` and treats every directive it prints as a hard requirement for the run (`SKILL.md:12-20`). The loader is **self-locating** — it resolves the shared registry relative to `BASH_SOURCE`, not the caller's cwd (`load-active-cards.sh:5-9,20-22`) — and does **exact-token** phase matching so that `deep-plan` does *not* match `deep-plan-review` (`load-active-cards.sh:34-40`). Cards are human-gated: never edit a card or the skill to disable one; use `directives/toggle.sh <ID> off` (`SKILL.md:20`). Both `load-active-cards.sh` and `references/artifact-structure.md` are declared **byte-identical copies** shared across every deep-* skill (`load-active-cards.sh:8-9`; `artifact-structure.md:3` "copied **verbatim** … All copies must stay byte-identical … A series consistency sweep verifies this").
**Alternatives considered / rejected.** Cross-skill `import`/reference of shared files — rejected by the "standalone rule": "skills self-contain; only the shared registry is shared" (`load-active-cards.sh:9`), and "the series standalone rule forbids cross-skill references" (`artifact-structure.md:3`). Baking learned improvements into each `SKILL.md` — rejected ("Improvements are data, never SKILL.md edits", `DESIGN-OUTLINE.md:5`).
**Why this choice.** Skills are distributed/installed independently, so each must be self-contained — hence verbatim copies plus an automated consistency sweep rather than a shared import. Self-location lets the same script work "no matter where the plugin is installed and no matter what the caller's working directory is" (`load-active-cards.sh:5-7`). Exact-token matching prevents the `deep-plan` / `deep-plan-review` prefix collision from leaking the wrong phase's cards. Keeping improvements as toggleable data (loaded lazily, scoped per phase) means the learning loop can add or retire a check via the registry without ever editing — or destabilizing — the skill itself.
